## What it is

Picture a friend who's an architect. You tell them "I want a nice house with 3 bedrooms." They ask you questions: "What color? How many windows? With a garden?" Then they build the whole house while you sip your coffee.

**Claude Web Builder** is that, but for landing pages. You answer questions about your business, and Claude builds **everything**: design, animations, SEO, deploy to Vercel.

Repo: `Hainrixz/claude-webkit`. Open source.

| Feature | Detail |
|----------------|---------|
| Code required | Zero |
| Language of the questions | Spanish |
| Stack it uses | Next.js 15 + Tailwind 4 + shadcn/ui + TS + Framer Motion |
| Deploy | Automatic Vercel |
| Skills included | 13 pre-installed |
| Total time | 10-30 min |

## The 6 automated phases

| Phase | What happens | Your job |
|------|----------|------------|
| 1 — Questions | Claude interviews you (4 rounds) | Answer in Spanish |
| 2 — Design | Shows you a visual plan | Approve or adjust |
| 3 — Build | Generates components | Watch (optional) |
| 4 — Preview | You see the page in a browser | Review |

## The 6 phases in detail

### Phase 1 — Discovery questions

4 rounds of simple questions:

**Round 1 — Business**:
- What do you do?
- Who is it for?
- What's your unique value proposition?

**Round 2 — Brand**:
- What tone do you want? (professional, friendly, technical, playful)
- What colors represent you?
- Do you have a logo?

**Round 3 — Content**:
- What sections do you need?
- Do you have copy or should Claude generate it?
- Main CTA?

**Round 4 — Technical**:
- Subdomain or your own domain?
- Analytics? (GA4, Plausible)
- Does the form go to a specific endpoint?

### Phase 2 — Design approval

Before coding, Claude shows you the **plan**:
- Proposed color palette
- Typography pairing
- Overall style (clean / bold / editorial)
- Section structure

If you like it → you continue.
If not → "swap X for Y" → it adjusts → shows a new plan.

### Phase 3 — Automatic build

Claude starts generating:
- Header / Navbar
- Hero with main CTA
- Services/features section
- Testimonials or proof
- Pricing (if applicable)
- FAQ
- Footer

You watch it get built **in real time** in the editor. Every component comes out complete and consistent.

### Phase 4 — Local preview

```bash
npm run dev
```

You open `localhost:3000` and see your full page:
- Responsive (try mobile/tablet/desktop)
- Animations working
- Forms (dev mode)

### Phase 5 — Iterative refinement

If something isn't convincing:

```
> Change the main CTA color to green
> Move the testimonials section above pricing
> Reduce the hero padding
```

Claude adjusts it instantly. Iterate as many times as you want.

### Phase 6 — Deploy to Vercel

```
> Deploy to Vercel
```

Claude:
1. Verifies the build works
2. Creates a Vercel account if you don't have one (it guides you)
3. Configures the project
4. Deploys
5. Hands you a public URL

**Your page ends up live on the internet with a real URL**. Ready to share.

I need a landing that covers:
- My value proposition
- Past cases

## Anti-patterns

### 1. Answering the questions vaguely

❌ "A nice landing"
✅ A specific brief with audience, tone, constraints

The discovery questions are **the most important part** of the flow. If you answer weakly, the output is weak.

### 2. Not iterating

The first version is 80%. The final 20% comes from specific feedback. **Don't settle for the first one** if it's not what you want.

### 3. Skipping the preview

Before deploying, **always** look at the preview. Sometimes it looks perfect on desktop but broken on mobile. Validate all 3 breakpoints.

### 4. Deploying without checking the form

The contact form comes out **functional but wired to console.log** by default. If you're going to use it for real, configure the endpoint **before** going public.

### 5. Wanting complex multi-page

Web Builder is designed for a **landing**. If you need a multi-page site with lots of routes (blog, internal app, dashboards), you need a different approach.
