## The problem

Everyone who builds pages with Claude Code or Cursor suffers the same thing:

- Ask for a landing → you get flat shadcn
- Ask for a dashboard → you get flat shadcn
- Ask for a store → you get the same boring template as always

**It's not Claude**. It's that without a library of premium pieces and a rules file, it defaults to flat shadcn. And flat shadcn is nice but **everyone already has it**.

**Cult UI** is 75 open-source premium components + 2 official skills for Claude that make it follow the spec without reminders.

Repo: `nolly-studio/cult-ui`. ~4.2k stars. MIT.

## What almost nobody knows — the 2 official skills

Here's the secret. Beyond components, the repo ships **2 official skills** for Claude Code that live in `.claude/skills/`.

### Skill 1 — `components-build`

16 rules Claude follows when it builds/refactors UI components:

- accessibility
- as-child
- composition
- data-attributes
- definitions
- design-tokens
- documentation
- marketplaces
- npm
- overview
- polymorphism
- principles
- registry
- state
- styling
- types

**Co-authored by Hayden Bleasel + shadcn**. Same spec that nolly-studio and shadcn use internally to keep things consistent.

### Skill 2 — `fixing-motion-performance`

An animation performance auditor. 9 categories of rules. Slash command `/fixing-motion-performance` to audit a specific file.

Detects:
- Layout reads/writes in the same frame
- Animating width/height/top/left (instead of transform/opacity)
- Misused `will-change`
- setInterval vs requestAnimationFrame
- Throttling of scroll/resize

### Why they change everything

Once downloaded to your project's `.claude/skills/`, **Claude reads them on its own at startup**. It follows the spec without you ever reminding it.

Without the skills: Claude defaults to its own patterns.
With the skills: Claude **respects the spec automatically**.

## Required stack

| Dependency | Version | Note |
|-------------|---------|------|
| **Tailwind CSS** | v4 (NOT v3) | Cult UI uses the v4 PostCSS plugin. On v3 the components come out unstyled |
| **motion** | latest | NOT `framer-motion`. Framer Motion was renamed to `motion` |
| **shadcn CLI** | `@latest` or `@beta` | Direct install with a URL works with @latest. The `@cult-ui/<name>` syntax requires @beta |
| **Next.js** | 14+ recommended | Also works on Vite or Astro |
| **clsx + tailwind-merge** | latest | For the `cn()` helper |

## Full setup — a single prompt

Paste this into Claude Code from the project root. It verifies the stack, installs missing dependencies, configures the registry, and adds the first component as a smoke test.

```
I want to use Cult UI (https://cult-ui.com · github.com/nolly-studio/cult-ui)
in this project. It's a shadcn registry, not an npm package. Do it in 5
steps, confirming each one:

1. VERIFY STACK
   - Read package.json: Tailwind version, framework, whether shadcn is present
     (components.json), whether motion or framer-motion is installed
   - If Tailwind is v3, propose an upgrade to v4 (do NOT run it yet)
   - If I have framer-motion: flag it. Cult UI uses motion, NOT framer-motion

2. INITIALIZE SHADCN IF MISSING
   - If there's no components.json: npx shadcn@latest init with reasonable
     defaults (neutral palette, CSS variables, neutral base color)
   - If it already exists: don't touch it

3. INSTALL MISSING DEPENDENCIES
   - motion (NOT framer-motion). If I have framer-motion, uninstall it first
   - clsx and tailwind-merge if missing

4. CONFIGURE THE @cult-ui REGISTRY
   - Edit components.json and add the block:
     "registries": {
       "@cult-ui": "https://cult-ui.com/r/{name}.json"
     }
   - If it already had a registries block: merge without overwriting

5. SMOKE TEST
   - npx shadcn@latest add https://cult-ui.com/r/cosmic-button.json
   - Confirm the file ended up in components/ui/
   - Import it into a test page to verify it compiles

If it fails:
- "registry not found": use the direct URL instead of @cult-ui/
- motion/react imports fail: verify you do NOT have framer-motion
- Tailwind classes don't apply: confirm v4 with @tailwindcss/postcss

At the end tell me: what got installed, files changed, and the command
to add the next component.
```

are read at startup. From that moment on Claude respects the spec
automatically.

## The daily flow

You're working on a section. You spot something generic. You paste a prompt with the component + the section. Claude applies it while respecting the branding.

### Prompt — specific component

```
I want to add [COMPONENT] from Cult UI in [WHERE].

1. Add the component: npx shadcn@latest add https://cult-ui.com/r/[COMPONENT].json

2. Read my globals.css and tailwind.config to extract the real design tokens
   (palette, font-family, radius, spacing). Respect those tokens.

3. Import it into [WHERE]:
   - Replace the current equivalent element with the Cult UI version
   - Keep the copy/text content intact
   - Adjust props so it respects MY palette and MY font — don't leave me
     the repo's demo style
   - If it needs a wrapper to animate (Suspense, client-only dynamic import),
     add it correctly

4. Verify:
   - Build/dev with no warnings
   - The component is integrated, NOT slapped on like a sticker

If I have the components-build skill loaded, follow it to the letter
(accessibility, polymorphism, project design-tokens NOT hardcoded).

At the end: files touched, before/after in 1 sentence, estimated FPS of
the animations. If under 60, suggest /fixing-motion-performance.
```

### Prompt — audit a whole page

```
Audit [PATH, e.g. app/page.tsx] and tell me where Cult UI would add value.

Your task is to AUDIT, not refactor. Produce this report:

1. INVENTORY
   - Key visual elements (hero, CTAs, cards, nav, footer, backgrounds)
   - For each: which shadcn component/raw HTML I use today

2. DIAGNOSIS
   - Flag which ones feel generic / template-y
   - Flag which ones are already good and should NOT be touched

3. PROPOSAL WITH CULT UI
   - For each generic one: the specific Cult UI component that replaces it
   - Only components you know exist in the registry
   - For each proposal: the exact install command
   - Visual impact: high/low

4. PRIORITIZED PLAN
   - Order from highest to lowest impact
   - Recommend starting with 2-3 high impact ones above the fold

5. RISKS
   - Will anything clash with my palette/typography?
   - Does any component require extra setup (motion, shader support)?

Don't add or install anything yet. Just the report.
```

## Common mistakes + fix

| Symptom | Cause | Fix |
|---------|-------|-----|
| `npx shadcn add` fails with "project not initialized" | Missing `components.json` | `npx shadcn@latest init` first |
| Component copies but doesn't animate | Missing `motion` | `pnpm add motion`. If you have `framer-motion`, uninstall it first |
| Tailwind classes don't apply | You're on v3 | Migrate to Tailwind v4 |
| `@cult-ui/<name>` doesn't resolve | You're using `shadcn@latest` | Switch to `@beta` or use the direct URL |
| Claude keeps building generic UI | You didn't download the skills | Run the skills download prompt |

## Anti-patterns

### 1. Installing Cult UI without the skills

Without the 2 official skills, Claude **goes back to its defaults** and ends up ignoring the premium components. The skills are **what changes the game**.

### 2. Trying to combine 75 components

5-6 are **enough** for a premium landing. More is visual noise.

### 3. Mixing `framer-motion` with `motion`

Imports break. **Only one**. If you have framer-motion, uninstall it.

### 4. Hardcoding the repo demo's colors

The Cult UI demo uses specific colors. If you copy them literally, **you don't respect your brand**. Ask Claude to **respect `globals.css`** explicitly.

### 5. Not running `/fixing-motion-performance`

Animations that look nice on desktop can be **catastrophic on mobile**. Validate performance before deploy.

| Case | Why |
