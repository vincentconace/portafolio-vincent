## Who it's for

You don't need prior technical experience. Just:
- A laptop
- The desire to experiment
- Clarity to describe what you want to create

**We're not trying to turn you into an engineer in an afternoon**. We want you to build something real and **put it on the internet TODAY**.

## One-line setup

```bash
# Mac / Linux
curl -fsSL https://raw.githubusercontent.com/construyeconia/setup/main/install.sh | bash

# Windows PowerShell
irm https://raw.githubusercontent.com/construyeconia/setup/main/install.ps1 | iex
```

This installs:
- Codex CLI
- Base dependencies (git, node, npm)
- Starter project
- Local server
- cloudflared tunnel for a public URL

5-10 min later you're ready to build.

## The flow — 4 phases

### Phase 1 — Define your idea (15 min)

Before touching Codex, answer:
- **What your app does** (1 line)
- **Who it's for** (audience)
- **What's the MOST important** thing that has to work
- **What it looks like** (visual reference if you have one)

If you struggle to answer, use the [VibeCoding 101](/boveda/vivecoding-101) prompt for a PRD.

### Phase 2 — Build the MVP (1-3h)

```
> I want to build [DESCRIBE YOUR APP].

  Brief:
  - Audience: [...]
  - MUST-have features: [3 max]
  - Visual style: [reference]
  - Stack: whatever works well

  Build me a working MVP running locally.
  Show me when it's ready to test.
```

Codex generates code, shows you a preview, you iterate.

### Phase 3 — Iterate (30-60 min)

You test it. Something's off → be specific:

```
> The Hero button is too small, make it prominent.
> The background color is too dark, make it lighter.
> The email form doesn't send a confirmation.
```

Codex adjusts.

### Phase 4 — Publish (15-30 min)

```
> It's ready. Publish it:
  1. Create a repo in my GitHub
  2. Deploy to Vercel
  3. Configure the domain
  4. Give me the final public URL
```

URL in hand.

## Anti-patterns

### 1. Wanting Spotify in v1

Your first app **doesn't need features from established products**. 3 well-built features > 15 mediocre ones.

### 2. Polishing something ugly

If you polish for 1 week, you lose momentum. **Ship it ugly and improve it later**.

### 3. Not publishing "because it's not perfect"

Perfect = you never publish it. **Publish today. Iterate this week**.

### 4. Switching stacks halfway

You started Codex with Next.js. **Don't switch** to SvelteKit halfway. Stick with it.

### 5. No real feedback

Show the URL to 5-10 people. Their feedback **is worth more than your opinion**.
