## Qué es

El pitch de agent-sh es claro: "AI escribe código. Esto automatiza todo lo demás." Y de hecho la mayoría de los plugins del marketplace son sobre el "todo lo demás" — quality gates, ship, audits, debate entre herramientas, onboarding, performance.

Es un marketplace de 19 plugins, con cross-compatibilidad probada: funciona en Claude Code, OpenCode, Codex, Cursor y Kiro. No te ata a una sola herramienta.

El plugin estrella es `next-task`: un orquestador autónomo que arma flujos con optimización de modelo (Opus para pensar, Sonnet/Haiku para ejecutar), gestión de estado en dos archivos, gates de enforcement y 8 sub-agents especializados. Si querés un loop autónomo bien armado, no tener que hacerlo desde cero.

## Cómo se instala

```bash
/plugin marketplace add agent-sh/agentsys

# Después instalás los que necesites
/plugin install next-task@agent-sh-agentsys
/plugin install gate-and-ship@agent-sh-agentsys
```

O `/plugin` → Discover → tab "agent-sh".

Para Claude Code en lenguaje natural:

```
Instalá el marketplace agent-sh/agentsys. Después agregame next-task,
prepare-delivery y ship. Mostrame el flujo recomendado para usarlos juntos.
```

## Qué podés hacer con esto

### Loop autónomo hasta el ship

`next-task` toma una tarea, arma el plan, lo ejecuta con sub-agents, y pasa por gates antes de avanzar al siguiente paso. No es lo mismo que [/goal](/boveda/goal-claude-code) — `next-task` ya viene con gates de calidad armados, mientras que `/goal` es el primitivo nativo de Claude Code.

### Pre-flight check antes de cada PR

`prepare-delivery` corre en pipeline: detecta slop con regex y analyzers, simplifica código duplicado, valida configs con `agnix` (342 reglas), corre el review loop y sincroniza docs. Cuando termina, el código está listo para que `ship` lo mande a prod.

### Segunda opinión cross-tool

Si solo te confiás de Claude para revisar, perdés un canal. `consult` te permite pedirle a Gemini CLI, Codex CLI o Copilot CLI que revisen el output de Claude — y `debate` arma rondas estructuradas con proposer/challenger hasta consenso. Misma filosofía que [Claude + Codex en equipo](/boveda/claude-codex-equipo).

### Linter de tu propia config

`agnix` revisa tu `CLAUDE.md`, las skills locales, los hooks y los MCPs contra 342 reglas. Si tu config se fue de las manos (y todas se van), te dice qué arreglar.

### Patrones repetidos → automatizaciones

`skillers` mira tus sesiones, detecta lo que hacés a mano repetidamente, y te sugiere armar un skill/hook que lo automatice. Es el equivalente a un "promote to skill" inteligente.

## Beneficios reales

- **Cubre el gap entre "Claude escribió código" y "el código está en producción"** — la parte que la mayoría ignora.
- **Cross-tool de verdad** — si mañana querés probar Cursor o Codex, los plugins siguen sirviendo.
- **Gates de calidad explícitos** en `prepare-delivery` — no es un wishlist, es un pipeline con steps verificables.
- **Modular** — instalás 1, 2 o los 19, según necesidad.

## Cuándo NO conviene

<Callout type="warning" title="Antes de instalar">

- **Si recién instalaste Claude Code:** son 19 plugins con dependencias entre sí. Arrancá por el [setup oficial](/boveda/claude-code-setup) y un par de skills básicas.
- **Si tu workflow ya está cubierto:** muchos de estos plugins resuelven cosas que vos quizás ya tenés con CI/CD propio + Cursor + Codex. Sumarlos puede solapar.
- **Si no querés ejecuciones autónomas largas:** `next-task` está pensado para correr horas sin supervisión. Si esa idea te incomoda, no lo uses.
- **Costo de tokens:** los plugins de debate y review iterativo consumen mucho. Mirá tu cuota antes.

</Callout>
