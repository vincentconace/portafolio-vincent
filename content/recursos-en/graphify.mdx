## The problem — Claude burning tokens

You ask Claude Code to understand your repo. Here's what it does:

1. `Glob` to find relevant files → finds 200
2. `Read` the first one
3. `Read` the second one
4. `Read` the third one...

By file 30 you've already burned half your context and it still hasn't answered your question. In a 3-year-old repo, you don't even make it to the end.

**Every Read is real tokens that you pay for.**

For small questions it doesn't matter. For **architecture questions** ("how does the auth module connect to payments?") or **onboarding** ("explain the whole app to me"), the cost is huge.

---

## The idea of a "repo graph"

Instead of making Claude open files to understand structure, **pre-process the repo** once and store:

- **The map**: which files exist, how they import each other, what each one exports
- **The brain**: descriptions of what each module does, architecture decisions
- **The detective**: reverse dependencies — who uses what, blast radius of changes

All stored in 2 files: `GRAPH_REPORT.md` (readable) and `graph.json` (serialized for queries).

When Claude asks about architecture, it reads **these 2 files** instead of 200 loose files. **Typical reduction: 50-70× fewer tokens** on architecture queries.

---

## The 3 outputs

### 1. The map (`GRAPH_REPORT.md`)

Navigable markdown with:

- List of top-level modules
- For each module: what it exports, what it imports, line counts
- Diagram (ASCII or Mermaid) of the main relationships

Useful for **onboarding** — open this file and understand the project in 10 minutes.

### 2. The brain (`graph.json`)

Serialized structure with nodes and edges:

```json
{
  "nodes": [
    { "id": "lib/auth", "type": "module", "exports": [...], "summary": "..." },
    { "id": "lib/payments", "type": "module", "exports": [...], "summary": "..." }
  ],
  "edges": [
    { "from": "api/checkout", "to": "lib/payments", "type": "import" },
    { "from": "lib/payments", "to": "lib/auth", "type": "import" }
  ]
}
```

Claude queries it to answer specific questions without reading source code.

### 3. The detective — dependency analysis

Reports derived from the graph:

- **Blast radius**: if I change file X, what breaks?
- **Dead code**: modules that nobody imports
- **Hotspots**: files imported from tons of places (refactor risk)
- **Cycles**: circular dependencies

## Setup

### Installation

```bash
# Via uv (fast)
uv tool install graphify

# Or pipx
pipx install graphify

# Or classic pip
pip install graphify
```

### Generate the graph

From the project root:

```bash
graphify build .
```

It processes the repo and generates `GRAPH_REPORT.md` and `graph.json` in `./graphify/`.

Typical time:
- Small repo (~50 files): 30 seconds
- Medium repo (~500 files): 2-5 minutes
- Large repo (~5000 files): 15-30 minutes

### Connect it to Claude Code

Add this to the project's `CLAUDE.md`:

```markdown
## Project structure

Before exploring individual files to understand architecture,
**always** check:

- `./graphify/GRAPH_REPORT.md` — project map
- `./graphify/graph.json` — serialized relationships

The graph is pre-processed. Use it instead of doing multiple Reads.
```

Now when Claude has to answer something about architecture, it goes to the graph first.

---

## Supported languages

Graphify understands ~25 languages (the popular ones):

- **Web**: TypeScript, JavaScript, Python, Ruby, PHP
- **Systems**: Go, Rust, C, C++, Java, Kotlin, Swift
- **Data**: R, Julia
- **Others**: C#, Scala, Elixir, Haskell, Lua, Perl

For each one it understands **imports/requires** and **exports/declarations**. Quality varies by language (TS/Python are better supported than Lua).

## Real use cases

### Case 1 — Onboarding to a legacy project

You get assigned a project that no current dev wrote. 3 years of code, no docs.

```bash
graphify build .
```

```
> Read ./graphify/GRAPH_REPORT.md and explain to me:
  1. What is the "core" module that everything else depends on?
  2. What are the 3 most independent folders (testable in isolation)?
  3. Is there any circular dependency that signals technical debt?
  4. Which modules look "legacy" (nobody imports them)?

  Summarize it in 1 page.
```

In 5 minutes you have the mental map. **Without Graphify, this takes you 2-3 hours reading folders.**

### Case 2 — Refactoring with blast-radius awareness

```
> I want to rename the `validateOrder` function in lib/orders/validate.ts.
  Before doing it:

  1. Check graph.json to see ALL the files that import it
  2. Make a list of the necessary changes
  3. Estimate the blast radius (small / medium / large)
  4. If it's large, propose a gradual change strategy

  Do NOT start editing until I approve the plan.
```

Much safer than doing find/replace and praying.

### Case 3 — Quick audit for a client

You're a consultant. They hand you a repo, you have 1h to diagnose it:

```bash
graphify build /path/to/client-repo
```

```
> I'm a consultant reviewing this repo. Generate a 2-page executive
  report:

  1. Overall health: organization, modularity, visible technical debt
  2. Top 5 structural problems
  3. Top 3 quick-win opportunities
  4. Recommendation for deeper changes (if any)

  Base it mainly on the graph, don't read the code.
```

A professional report in minutes.

### Case 4 — PDFs and papers as a navigable course

Graphify isn't just for code. Throw a folder of PDFs at it:

```bash
graphify build ./client-docs/
```

It processes the PDFs, identifies concepts, and generates a graph of references between documents.

Then:

```
> The client gave us 40 PDFs about their platform. Read the graph and
  propose a reading order that prioritizes the central documents
  first (the ones most referenced).
```

You turn 40 unordered PDFs into a coherent sequential course.

## Anti-patterns

### 1. Building the graph and not using it

If you generate it but don't add to `CLAUDE.md` that Claude should **use it first**, Claude will keep reading files. **The instruction in CLAUDE.md is critical.**

### 2. Not regenerating after big changes

The graph is a snapshot. If you refactor major structure, the graph goes stale. **Rebuild periodically** or whenever the repo changes significantly.

### 3. Expecting it to replace reading code

The graph is for **structural navigation**, not for understanding fine-grained logic. For "what exactly does this function do", Claude has to read the code. **It's a complement, not a replacement.**

### 4. Trusting the blast radius 100%

Static analysis doesn't capture everything (dynamic imports, monkey patching, eval). The blast radius is a **good approximation**, not absolute truth.

## Comparison with alternatives

| Approach | Token cost | Speed | Precision |
|----------|--------------|-----------|-----------|
| **Claude reads file by file** | High | Slow | High (but only what was read) |
| **Graphify + graph** | Low (50-70× less) | Fast | Good for structure, not for logic |
| **RAG with embeddings** | Medium | Medium | Good for semantic search |
| **Manual: document by hand** | Zero | Very slow | Best, but costly to maintain |

For **architecture questions, blast radius, onboarding**: Graphify wins.
For **semantic search in docs**: RAG (Book-to-Skill, embeddings) wins.
For **understanding the detailed logic of a function**: reading directly wins.

The 3 approaches are **complementary**, not substitutes.
