## Qué es

Para teams con spec-driven development: antes de editar production code, debería haber un `.spec.md` documentando el cambio. Este hook verifica que exista uno reciente (default: últimos 14 días) y warn si no.

Hook event: `PreToolUse`.

Encaja con la filosofía de [Spec Kit](/boveda/spec-kit) y workflows similares.

## Cómo se instala

```bash
npx claude-code-templates@latest --hook quality-gates/plan-gate
```

Para Claude:

```
Instalé el hook quality-gates/plan-gate. Mostrame los .spec.md
existentes y cuáles entran dentro de los últimos 14 días. Si edito
prod code sin spec reciente, debería warn.
```

## Qué podés hacer con esto

### Enforce spec-first development

Cualquier cambio en prod requiere un spec aprobado. El hook lo enforce sin tener que recordarlo.

### Combo con [Spec Kit](/boveda/spec-kit)

Spec Kit genera y mantiene specs. Plan Gate enforce que existan. Combo natural.

### Para teams enterprise

Compliance y traceability: cada cambio en prod tiene su spec documentado y auditable.

### Para sesiones autónomas

Claude con `/goal` no toca prod sin spec. Red de seguridad para autonomous workflows.

## Cuándo NO conviene

<Callout type="warning" title="Antes de instalar">

- **Si tu equipo no usa spec-driven:** el hook agrega fricción.
- **Para repos personal / experimental:** specs son overhead.
- **Si tu definición de "prod code" no es clara:** el hook puede ser ambiguo.
- **Sin Spec Kit u otra fuente de specs:** vas a crear .spec.md a mano cada vez.

</Callout>
