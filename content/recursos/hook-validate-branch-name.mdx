## Qué es

Antes de cualquier `git checkout -b` o `git switch -c`, valida que el nombre de la branch siga Git Flow:
- `feature/<name>` para features
- `release/v*.*.*` para releases (semver)
- `hotfix/<name>` para hotfixes

Si no matchea, bloquea la creación. Hook event: `PreToolUse`.

## Qué podés hacer con esto

### Enforce Git Flow sin disciplina

Vos no tenés que recordar la convention — el hook la enforce.

### Para teams donde branch name matters

CI scripts que matchean por prefix (`feature/*` triggerea ciertos tests), release tooling automático — todo necesita convention respetada.

### Para sesiones autónomas

Claude no inventa nombres random — debe seguir la convention.

### Combo con [Prevent Direct Push](/boveda/hook-prevent-direct-push) y [Conventional Commits](/boveda/hook-conventional-commits)

Triple gate Git Flow:
1. Branch name valid
2. No direct push a main
3. Conventional commit format

## Cuándo NO conviene

<Callout type="warning" title="Antes de instalar">

- **Si tu team no usa Git Flow:** otro convention podría aplicar (trunk-based, etc.). Ajustá patterns o no instales.
- **Para repos personal / experimental:** overhead.
- **Si tu CI ya valida:** este hook duplica pero local.
- **Quick prototypes:** override puede ser frecuente.

</Callout>
