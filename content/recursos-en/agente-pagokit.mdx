By the end of this resource, you'll have:

- A **working checkout** integrated into your project, ready to receive real payments
- Your **chosen payment provider** configured based on your customers' profile
- **5 security locks** implemented automatically (server-side validation, webhook signing, etc.)

What normally takes 2-3 days of integration, this plugin does in a single command.

## Installation in 3 steps

### Step 1 — Add the plugin to Claude Code

<Step number={1} title="Install the plugin from GitHub">

This command downloads the plugin and registers it in your local Claude Code installation. The plugin includes the agent's full context: which questions to ask, how to generate the code, and how to validate that the integration works.

**Why this matters:** Claude Code without plugins knows nothing about your payment stack. The plugin gives it specific instructions about Stripe, Mercado Pago, Wompi and Lemon Squeezy — including the differences between their APIs and the common mistakes to avoid.

```bash
claude plugins install github:Hainrixz/agente-pagokit
```

After this command, you should see:

```
✓ Plugin "agente-pagokit" installed successfully (v1.0)
✓ Available commands: configure-payment, test-payment, check-security
```

If you see a permissions error, run it with `sudo` or check that you have write access to Claude's plugins directory.

</Step>

### Step 2 — Configure your provider

<Step number={2} title="Run the interactive configurator">

This is the main command. Claude will ask you exactly 3 questions and then generate all the integration code.

**Why only 3 questions:** The agent is designed not to overwhelm you with options. Those 3 questions (customer location, billing model, and payment methods) determine 90% of the architecture decisions. The remaining 10% is resolved with the safest defaults.

```bash
claude configure-payment
```

The agent will present the interactive flow, generate the files, and at the end show a summary of what it created and the environment variables you need to configure.

</Step>

### Step 3 — Add the environment variables

<Step number={3} title="Configure the API keys" last>

The agent generates a `.env.example` file with all the variables you need. Copy this file to `.env.local` (Next.js) or `.env` (Express/Remix) and fill in the values with your real keys.

**Why it doesn't ask for them in the command:** API keys should never pass through the command line — they'd stay in the terminal history. The `.env` file is in `.gitignore` by default, so it's the right place.

```bash
# Copy the template
cp .env.example .env.local

# Open the file and fill in the values
# (use your favorite editor)
```

Example of what you'll see in `.env.example`:

```bash
# Generated by PagoKit Agent v1.0
# NEVER commit this file to Git

# Your chosen provider
PAYMENT_PROVIDER=mercado-pago

# Mercado Pago
MP_PUBLIC_KEY=APP_USR-xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx
MP_ACCESS_TOKEN=APP_USR-0000000000000000-000000-xxxxxxxxxxxxxxxxxxxx
MP_WEBHOOK_SECRET=your_webhook_secret_here

# Your app URL
NEXT_PUBLIC_APP_URL=https://your-domain.com
```

</Step>

## The agent's 3 questions

When you run `claude configure-payment`, the agent asks exactly these questions. Here we explain the reasoning behind each one, so you make the right decision.

### Question 1: Where are your customers from?

This question determines which provider has the best coverage and the lowest fees for your market.

If your customers are from **Latin America**, using Stripe directly means many payments with local cards (especially debit) will fail or carry extra fees. Mercado Pago has direct contracts with all LATAM banks.

If your customers are **international**, Stripe is the standard. Its infrastructure covers 135+ countries with a single contract.

<Callout type="warning" title="Common mistake">
Using Stripe for Mexican/Argentine customers because "it's better known." Rejection rates with local debit cards can reach 30-40% with Stripe. With Mercado Pago, those same cards have approval rates of 85-95%.
</Callout>

### Question 2: One-time payment or subscription?

This question determines the backend architecture.

A **one-time payment** is simple: create payment intent → confirm → webhook → mark as paid. 4 steps.

A **subscription** is different: create customer → create subscription → handle renewals → handle cancellations → handle failed payments → retries → dunning. 20+ events to handle.

The agent generates different database structures for each case. If you start with the one-time payment architecture and then want to switch to subscriptions, it's an almost complete rewrite.

<Callout type="tip" title="If you don't know yet">
If your business model isn't clear yet, choose "subscription" even if you start with one-time payments. The subscription architecture can handle one-time payments (a subscription with indefinite duration), but not the other way around.
</Callout>

### Question 3: Digital only or cash too?

In Latin America, a significant part of the adult population doesn't have a credit card. In Mexico, 54% of e-commerce transactions include some cash component (OXXO Pay, CoDi, etc.).

If you choose **digital only**, the flow is synchronous: the payment confirms in seconds.

If you choose **cash too**, the flow is asynchronous: the customer generates a code, pays at a store/bank, and hours later the confirmation webhook arrives. This requires a different UI ("waiting for payment confirmation") and state handling in the database.

The agent generates both flows correctly, but it needs to know which one to implement.

## The 4 providers explained

### Quick comparison

| Provider | Best for | Cash | Approx. fee | Subscriptions |
|---|---|---|---|---|
| **Mercado Pago** | All of LATAM | ✓ OXXO, Rapipago, etc. | 3.49% + VAT | ✓ Basic |
| **Wompi** | Colombia | ✓ PSE, Nequi | 2.95% | ✓ |
| **Stripe** | International | ✗ | 2.9% + $0.30 | ✓ Advanced |
| **Lemon Squeezy** | Digital products | ✗ | 5% + $0.50 | ✓ + License keys |

### Mercado Pago

The LATAM payment giant. If your market is Mexico, Argentina, Colombia, Chile, Brazil, Peru, Uruguay or Bolivia, this is the option with the most coverage.

**Real advantages:**
- Accepts local methods Stripe doesn't handle: OXXO, Rapipago, Pago Fácil, PSE, Efecty, Baloto
- Automatic currency conversion (you charge in USD, the customer pays in local pesos)
- Approval rates 15-20% higher than Stripe for local cards

**Important consideration:** The Mercado Pago dashboard has a learning curve. Its webhooks also have particularities (topic, resource_id) that differ from Stripe. The agent handles this, but if you want to understand the generated code, it's worth reading their documentation.

### Wompi

The best option for Colombia specifically. It integrates directly with PSE (the Colombian bank debit system), Nequi (Bancolombia's digital wallet), and national cards.

**Why not use it outside Colombia:** Its international coverage is limited. If you have customers in other LATAM countries, you'll need a second provider anyway.

### Stripe

The industry standard for international payments. Its documentation is the best in the sector, and practically the entire ecosystem of SaaS tools (Supabase, Vercel, etc.) has native integrations with Stripe.

**When it's the right choice:**
- Customers in Europe, the US or Canada
- You need advanced billing (invoices, credits, programmatic discounts)
- You want guaranteed webhook reliability with automatic retries
- Your stack already uses other Stripe integrations (Radar, Connect, etc.)

### Lemon Squeezy

The "Merchant of Record" — they're legally the seller, which means they handle EU VAT, US sales tax, and tax compliance in 40+ countries for you.

**Who it's indispensable for:**
- You sell software, courses, ebooks, or any digital product
- You have customers in the EU or US where digital VAT/sales tax is mandatory
- You don't want to hire an international tax accountant

**Trade-off:** Their fee (5% + $0.50) is higher. But compared to tax fines of $10,000+ in the EU, it's a smart investment.

## The 14 generated files

The agent doesn't generate a single monolithic file. It generates a clean, separated architecture that you can maintain and scale.

### Visual summary

| File | What it does |
|---|---|
| `components/checkout/CheckoutForm.tsx` | Payment form with validation |
| `components/checkout/PaymentMethodSelector.tsx` | Payment method UI selector |
| `components/checkout/OrderSummary.tsx` | Order summary before paying |
| `components/checkout/SuccessPage.tsx` | Post-payment confirmation page |
| `app/api/payments/create-intent/route.ts` | Creates the payment intent with the provider |
| `app/api/payments/confirm/route.ts` | Confirms the payment from the client |
| `app/api/webhooks/payment/route.ts` | Receives and processes provider events |
| `app/api/webhooks/payment/handlers.ts` | Logic for each event type |
| `lib/payments/provider.ts` | Provider abstraction (easy to switch) |
| `lib/payments/validation.ts` | Server-side validations |
| `lib/payments/errors.ts` | Payment error handling |
| `hooks/useCheckout.ts` | React hook for checkout state |
| `types/payments.ts` | TypeScript types for the entire integration |
| `.env.example` | Environment variables template |

### Why this architecture

The separation in `lib/payments/provider.ts` is intentional: if one day you need to switch from Mercado Pago to Stripe (or vice versa), you only modify that file. The rest of the code doesn't know which provider you're using.

## The 5 security locks

These are not optional. Each one prevents a real attack vector that happens in production.

### Lock 1: Server-side validation

**The problem it prevents:** An attacker can modify the price in the frontend (using the browser's DevTools) before submitting the form. If you only validate in the frontend, they can buy a $100 product by paying $1.

**How it works:** The `validation.ts` file verifies that the amount received by the API matches the real price stored in your database, not what the client sent.

### Lock 2: Webhook signature verification

**The problem it prevents:** Anyone can POST to your webhook endpoint pretending to be Mercado Pago or Stripe. Without verification, your system would mark orders as paid without anyone having paid.

**How it works:** Each provider includes a cryptographic signature in the webhook header. The agent generates code that verifies this signature using your webhook secret before processing any event.

<Callout type="danger" title="Critical">
This lock is the most important one. Without it, anyone with your webhook URL can confirm fake payments. The agent always implements it by default.
</Callout>

### Lock 3: Webhook idempotency

**The problem it prevents:** Payment providers resend webhooks if your server doesn't respond in time. Without idempotency, the same payment would be processed multiple times (double delivery, double balance credit, etc.).

**How it works:** The `handlers.ts` file stores the ID of each processed webhook. If the same webhook arrives twice, the second one is ignored.

### Lock 4: Rate limiting on payment endpoints

**The problem it prevents:** Brute-force attacks against cards (card testing attacks) — a bot tries thousands of card numbers until it finds one that works. This not only costs money in failed-attempt fees, but can result in your payment account being suspended.

**How it works:** The `create-intent` endpoint is limited to 10 attempts per IP per hour. If exceeded, it returns 429 Too Many Requests.

### Lock 5: Never expose the secret key to the client

**The problem it prevents:** If your `MP_ACCESS_TOKEN` or `STRIPE_SECRET_KEY` reaches the frontend (through a code error or Vercel's environment variables), an attacker can make charges to your account or extract customer data.

**How it works:** The agent explicitly separates environment variables into `NEXT_PUBLIC_*` (frontend, public) and no prefix (server only). During code review, it verifies that no sensitive variable has the `NEXT_PUBLIC_` prefix.

## Implementation playbook

Follow these steps in order. The order matters because each step depends on the previous one.

**Step 1 — Install and configure (15 min)**
Run `claude configure-payment`, answer the 3 questions, fill in the `.env.local`.

**Step 2 — Test locally with webhooks (10 min)**
Use ngrok or the Stripe CLI to expose your localhost and receive real webhooks in development. The agent generates the exact command for this.

```bash
# The agent includes this command in the output
stripe listen --forward-to localhost:3000/api/webhooks/payment
# or for ngrok:
ngrok http 3000
```

**Step 3 — Make a test payment (5 min)**
Each provider has test cards. The agent includes the most useful ones in the `TESTING.md` file it generates:

```
Stripe - Successful payment: 4242 4242 4242 4242
Stripe - Insufficient funds: 4000 0000 0000 9995
Mercado Pago - Approved: 5031 7557 3453 0604
```

**Step 4 — Verify the 5 locks (10 min)**
```bash
claude check-security
```
This command analyzes the generated code and verifies that the 5 locks are implemented correctly.

**Step 5 — Deploy and configure webhooks in production**
In your provider's dashboard, add the production URL for webhooks: `https://tu-dominio.com/api/webhooks/payment`

## When NOT to use this plugin

PagoKit Agent solves 80% of use cases. These are the cases where you need a custom implementation:

**Marketplaces with split payments:** If you have multiple sellers and need to distribute the payment automatically (Stripe Connect or Mercado Pago Marketplace), the agent doesn't cover that flow yet.

**Cryptocurrency payments:** Not included. For this, consider Coinbase Commerce or BTCPay Server.

**Legal electronic invoicing (CFDI Mexico):** The agent generates the checkout, but it doesn't generate the signed CFDI XMLs required by the SAT. For that you need an additional integration with Facturama or similar.

**Checkout embedded in mobile apps:** The generated code is for web. For React Native or Flutter, the providers' SDKs have specific mobile implementations.

**Systems with PCI DSS Level 1 compliance:** If you handle more than 6 million transactions per year, you need a full PCI certification that goes beyond what this plugin covers.
