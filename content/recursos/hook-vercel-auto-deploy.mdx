## Qué es

Después de cada commit, el hook dispara automáticamente un deploy a Vercel. Soporta:
- **Environment-specific strategies** (dev → staging → prod según branch)
- **Rollback on failure** automático

Hook event: `PostToolUse`.

## Cómo se instala

```bash
npx claude-code-templates@latest --hook automation/vercel-auto-deploy
```

Pre-requisitos:
- Vercel CLI instalado y autenticado
- Proyecto vinculado (`vercel link`)
- Tokens configurados

Para Claude:

```
Instalé el hook automation/vercel-auto-deploy. Verificá que tengo
vercel CLI autenticado. Hacé un commit y debería disparar deploy
con la strategy de mi branch.
```

## Qué podés hacer con esto

### Deploy continuo sin tocar dashboard

Commit → deploy. Sin pasos manuales.

### Strategies por branch

`main` → producción. `develop` → staging. `feature/*` → preview.

### Rollback automático

Si el deploy falla healthchecks, rollback al deployment anterior. Mitigación rápida.

### Combo con [Vercel Environment Sync](/boveda/hook-vercel-environment-sync)

Auto Deploy publica. Env Sync mantiene env vars consistentes entre local y Vercel.

### Combo con [Deployment Health Monitor](/boveda/hook-deployment-health-monitor)

Auto Deploy lanza. Health Monitor observa post-deploy.

## Cuándo NO conviene

<Callout type="danger" title="Antes de auto-deploy a producción">

- **Sin CI tests primero:** auto-deploy sin tests = manda código roto a prod. Combinalo con [Run Tests After Changes](/boveda/hook-run-tests-after-changes).
- **Para repos sin Vercel:** el hook es Vercel-only. Para Netlify, Railway, otros, alternativas.
- **Si no querés auto-deploy:** override siempre que necesités review humano.
- **Tokens con scope amplio:** un Vercel token con acceso a producción es serio. Restringí.

</Callout>
