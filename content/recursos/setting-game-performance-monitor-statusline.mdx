## Qué es

Statusline custom específica para game development. Muestra:
- **FPS targets** y compliance
- **Draw calls** count
- **Memory usage** en runtime
- **Polygon count** y build optimization metrics

Nicho: Unity, Unreal, Godot devs que viven con estos métricas como rutina.

```bash
npx claude-code-templates@latest --setting statusline/game-performance-monitor-statusline
```

```
Instalé el setting statusline/game-performance-monitor-statusline.
Mostrame las métricas que detecta en mi proyecto Unity/Unreal/Godot.
```

## Qué podés hacer con esto

### Métricas siempre visibles

Para game dev, métricas como FPS y draw calls son críticas. Tener en statusline siempre les da awareness.

### Build optimization tracking

Mientras codeás features, ves si las cambios afectan los métricas target.

### Combo con [Asset Pipeline Statusline](/boveda/setting-asset-pipeline-controller-statusline)

Asset Pipeline para procesamiento de assets. Game Performance para runtime. Cubren ángulos distintos del game dev workflow.

## Cuándo NO conviene

<Callout type="warning" title="Antes de instalar">

- **Si no hacés game dev:** las métricas no aplican.
- **Si tu engine no expone esas métricas via accessible interface:** el setting puede no detectarlas.
- **Para web devs:** [Context Monitor](/boveda/setting-context-monitor) es la opción.
- **Solo 70 installs:** muy nicho. Validá implementación.

</Callout>
