## Qué es

El blocker básico [Dangerous Command Blocker](/boveda/hook-dangerous-command-blocker) matchea patterns directos: `rm -rf`, `dd if=`. Pero `bash -c 'rm -rf /'` evade el match simple.

Este hook cubre ese gap: detecta comandos destructivos dentro de wrappers:
- `sh -c "..."`
- `bash -c "..."`
- `python3 -c "..."`
- `node -e "..."`
- `perl -e "..."`
- `ruby -e "..."`

Inspect el código embebido buscando destructivos. Hook event: `PreToolUse`.

## Qué podés hacer con esto

### Cerrar gap del blocker básico

Si solo tenés [Dangerous Command Blocker](/boveda/hook-dangerous-command-blocker), un command via wrapper pasa. Este hook lo cubre.

### Para sesiones autónomas

Modelos AI a veces "inventan" formas creativas de hacer cosas — incluyendo destructivos via wrappers. Defense in depth.

### Combo natural

Triple red de seguridad:
1. [Dangerous Command Blocker](/boveda/hook-dangerous-command-blocker) — patterns directos
2. **Shell Wrapper Guard (este)** — wrappers
3. [AI Bash Guard](/boveda/hook-ai-bash-guard) — AI-driven catch variaciones

## Cuándo NO conviene

<Callout type="warning" title="Antes de instalar">

- **Si tu workflow legítimo usa wrappers heavy:** sysadmins, devops. Override puede ser frecuente.
- **Sin [Dangerous Command Blocker](/boveda/hook-dangerous-command-blocker) instalado:** el complement queda solo. Instalá ambos juntos.
- **Solo 76 installs:** validá implementación.

</Callout>
