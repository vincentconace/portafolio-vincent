## What it is

Before any Write tool call, the hook checks whether the target is a `.env*` file (`.env`, `.env.local`, `.env.production`, etc.). If it is, it blocks it.

Deterministic filtering, zero cost (no LLM involved). Hook event: `PreToolUse`.

It solves a concrete risk: Claude mistakenly rewrites your `.env` with made-up values, and you lose all your real credentials.

## How to install it

```bash
npx claude-code-templates@latest --hook security/env-file-protection
```

For Claude:

```
I installed the security/env-file-protection hook. Try it: ask Claude
to create or overwrite `.env.local` with a value — it should get blocked.
```

Any Claude Code session. Zero cost, high protection.

If you leave Claude running with `/goal`, this hook prevents it from overwriting secrets by accident.

### Combo with [Dangerous Command Blocker](/boveda/hook-dangerous-command-blocker) and [File Protection](/boveda/hook-file-protection)

## When it's NOT a good fit

<Callout type="warning" title="Before installing">

- **If your legitimate workflow modifies .env:** the hook blocks direct Write. For edits, use the Edit tool (it may not be blocked — verify the behavior).
- **If your env vars live somewhere else** (vault, AWS Secrets): the hook only protects .env files. Other patterns need other hooks.
- **For initial setup:** when you create a .env for the first time, the hook blocks it. Manually override or set it up before installing.

</Callout>
