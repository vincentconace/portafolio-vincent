## What it is

Before every `git commit` that Claude runs, this hook validates that the message follows the Conventional Commits format:

```
type(scope): description
```

Supported types: `feat`, `fix`, `docs`, `style`, `refactor`, `test`, `chore`, `ci`, `perf`, `build`.

If the message doesn't respect the format, the commit is blocked. Hook event: `PreToolUse`.


## What you can do with it

### Convention enforced across the whole repo

It no longer matters if Claude throws out an informal commit. The hook formats it or rejects it.

### Auto-generated changelog

With conventional commits, tools like `conventional-changelog` build the CHANGELOG automatically at release time.

### Automatic semantic versioning

`feat:` → minor bump. `fix:` → patch. `BREAKING CHANGE:` → major. Your CI can handle it.

### For teams new to the convention

Enforcing the pattern is the fastest way to adopt the convention.

## When it's NOT worth it

<Callout type="warning" title="Before installing">

- **If your team doesn't use the convention:** don't impose it — align with the team first.
- **For personal/experimental repos:** the convention is overhead. Commit however you like.
- **If your CI already validates:** husky + commitlint do the same thing in a traditional pre-commit hook.
- **Quick WIP commits:** the hook blocks them. For urgent hotfixes, you'll need an override.

</Callout>
