## Qué es

Pre-autoriza npm commands de desarrollo común:
- `npm run lint`
- `npm run test`
- `npm run build`
- `npm start`

Es el #2 más adoptado de permission settings (1,823 installs), solo detrás de [Allow Git Operations](/boveda/setting-allow-git-operations) en categoría similar. Razón: workflow JS/TS sin interrupciones es lo que casi todos queremos.

## Cómo se instala

```bash
npx claude-code-templates@latest --setting permissions/allow-npm-commands
```

Para Claude:

```
Instalé el setting permissions/allow-npm-commands. Listame las
npm ops pre-autorizadas. Si quiero agregar `npm run dev`, mostrame
cómo.
```

## Qué podés hacer con esto

### Workflow JS sin interrupciones

`npm run lint && npm run test && npm run build` corre sin "permitir" cada step.

### Para sesiones autónomas

`/goal` que incluye build + test: el setting evita confirmations.

### Combo con [Lint On Save Hook](/boveda/hook-lint-on-save)

Hook dispara lint automatic. Setting autoriza la ejecución sin pedir permiso. Pipeline silencioso.

### Combo con [Run Tests After Changes Hook](/boveda/hook-run-tests-after-changes)

Setting + hook = tests corren sin confirmation cada edit.

### Para proyectos polyglot

Si tu proyecto mezcla npm con otros tools, este setting cubre el lado JS/TS.

## Cuándo NO conviene

<Callout type="warning" title="Antes de habilitar">

- **Scripts custom destructivos:** si tu `npm run something` es `rm -rf node_modules && reinstall`, autorizarlo sin pensar puede ser cuestionable.
- **Sin scripts en package.json:** el setting no hace nada útil.
- **Si querés revisar cada step:** este setting saltea revisión.
- **Para proyectos no-npm:** Yarn/pnpm pueden necesitar setting separado o equivalente.

</Callout>
