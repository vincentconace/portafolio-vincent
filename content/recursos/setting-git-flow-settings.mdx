## Qué es

Configuración all-in-one para teams que adoptan Git Flow seriamente. Combina:

- **Statusline** custom mostrando Git Flow status en tiempo real
- **Permissions** (allow 21 + deny 6) tuned para Git Flow
- **Env vars** que definen prefixes y branch names:
  - `GIT_FLOW_MAIN_BRANCH: main`
  - `GIT_FLOW_DEVELOP_BRANCH: develop`
  - `GIT_FLOW_PREFIX_FEATURE: feature/`
  - `GIT_FLOW_PREFIX_RELEASE: release/`
  - `GIT_FLOW_PREFIX_HOTFIX: hotfix/`
  - `GIT_FLOW_VERSION_TAG_PREFIX: v`
- **Hooks PreToolUse** (3 hooks) para workflow enforcement

Es la opción "todo-armado" para enforce Git Flow consistente.

## Qué podés hacer con esto

### Git Flow enforcement consistente

Setting único enforce TODOS los aspectos: naming, permissions, statusline awareness.

### Onboarding de teams a Git Flow

En vez de explicar 4 cosas distintas, instalás este setting y todos lo usan.

### Statusline con Git Flow context

Ves en qué tipo de branch estás (feature/release/hotfix) y si estás en compliance.

### Combo con hooks individuales

Si querés más control, podés desactivar el bundle y usar hooks individuales:
- [Validate Branch Name](/boveda/hook-validate-branch-name)
- [Prevent Direct Push](/boveda/hook-prevent-direct-push)
- [Conventional Commits](/boveda/hook-conventional-commits)

### Compliance audit

Si tu org requiere Git Flow para releases, este setting lo enforce automáticamente.

## Cuándo NO conviene

<Callout type="warning" title="Antes de instalar">

- **Si tu equipo no usa Git Flow:** otro workflow (trunk-based, GitHub Flow) necesita setting distinto.
- **Si querés más granularidad:** hooks individuales dan más control que el bundle.
- **Si tus branch names son distintos:** ajustá env vars antes.
- **Para repos personales / experimentales:** Git Flow es overhead.

</Callout>
