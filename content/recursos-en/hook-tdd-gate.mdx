## What it is

Orthodox TDD says: write a test, see it go red, write code, see it go green, refactor. This hook enforces the first step: you can't edit production files unless the matching test exists.

Supported languages:
- `.cs` (C#)
- `.py` (Python)
- `.ts`, `.tsx`, `.js`, `.jsx` (TS/JS)
- `.go` (Go)
- `.rs` (Rust)
- `.rb` (Ruby)
- `.php` (PHP)
- `.java`, `.kt` (Java/Kotlin)
- `.swift` (Swift)
- `.dart` (Dart)

Hook event: `PreToolUse`.

## What you can do with it

### Strict TDD enforcement

No more "I'll write the test later." The hook enforces it upfront.

### For autonomous sessions

Long `/goal` runs: Claude won't introduce code without a test. Quality baseline guaranteed.

### Adopting TDD in new teams

Forcing the pattern is the fastest way to internalize it.

### Combo with [Run Tests After Changes](/boveda/hook-run-tests-after-changes)

TDD Gate enforces that tests exist first. Run Tests After validates that they stay green afterward.

### Combo with [Plan Gate](/boveda/hook-plan-gate)

Plan Gate enforces the spec. TDD Gate enforces tests. A quadruple quality net.

## When it's NOT a good fit

<Callout type="warning" title="Before installing">

- **If your team doesn't do TDD:** the hook generates constant friction.
- **For spike code or experiments:** strict TDD blocks speed.
- **For massive refactors:** the hook can be overly cautious.
- **Without a clear test-naming convention:** the hook may not find existing tests.
- **If your test runner doesn't exist:** frequent overrides.

</Callout>
