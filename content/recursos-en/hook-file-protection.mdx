## What it is

A hook that blocks edits to files in a list of "protected" paths. It typically covers:
- System files (`/etc/`, etc.)
- Critical configuration files (CI/CD configs, base .gitignore, etc.)
- Production code that's clearly marked

More general than [Env File Protection](/boveda/hook-env-file-protection) (which is just `.env*`). Configuration defines which paths are protected.

Hook event: `PreToolUse`.

## How to install it

```bash
npx claude-code-templates@latest --hook security/file-protection
```

For Claude:

```
I installed the security/file-protection hook. Show me the default list of
protected paths and how to add more (for example `prod/**`, `infra/**`).
```

## What you can do with it

### Configurable protected paths

A list of globs (`prod/`, `infra/`, `secrets/`, etc.). When Claude tries to edit anything in that list, it gets blocked.

### Defense in depth

Combined with [Dangerous Command Blocker](/boveda/hook-dangerous-command-blocker) (for shell commands) and [Env File Protection](/boveda/hook-env-file-protection) (for .env), you get triple coverage.

### For teams with dev/prod separation

Your repo has `apps/dev/` and `apps/prod/`. You protect `apps/prod/` so Claude never touches it without an explicit override.

### For autonomous sessions

If Claude is running in `/goal`, this hook is essential. Without it, a wrong decision can touch production.

## When it's not worth it

<Callout type="warning" title="Before installing">

- **If your legitimate workflow edits the protected paths:** constant overrides = friction. Adjust the list.
- **For shops without dev/prod separation:** less critical.
- **For personal repos:** it can be overhead.
- **If your list of paths is ambiguous:** document exactly what is protected.

</Callout>
