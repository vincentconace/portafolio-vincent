## What it is

A list of MCP servers that you do NOT want to activate — typical examples from the catalog:
- `web-scraper` (potential misuse)
- `system-admin` (very broad access)
- `network-tools` (network access)

The `disabledMcpjsonServers` setting takes an array of names. Even if they're defined in the project's `.mcp.json`, they still stay disabled.

Blacklist approach: most servers pass through, only specific ones get blocked.

## How to install it

```bash
npx claude-code-templates@latest --setting mcp/disable-risky-servers
```

Edit the list with the real servers you want to block:

```json
{
  "disabledMcpjsonServers": [
    "web-scraper",
    "system-admin",
    "network-tools",
    "any-other-you-want"
  ]
}
```

## What you can do with it

### Enterprise policy enforcement

Your org says "don't use scraping MCPs." The setting enforces it without relying on each dev.

### Block experimental MCPs

A new MCP in alpha that you want to avoid — add it to the list.

### For teams with a shared .mcp.json

If your repo defines MCPs in `.mcp.json`, the setting allows a defensive override.

### Combo with [Enable Specific Servers](/boveda/setting-enable-specific-servers)

Disable Risky is a blacklist (most pass through). Enable Specific is an allowlist (only the listed ones). Opposite strategies — pick based on your preference.

## When it's NOT worth it

<Callout type="warning" title="Before installing">

- **If you don't use MCPs:** it adds nothing.
- **If your legitimate workflow needs the blocked servers:** constant overriding.
- **For a "deny by default" approach:** [Enable Specific](/boveda/setting-enable-specific-servers) is more restrictive.

</Callout>
