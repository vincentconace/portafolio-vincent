## What it is

Think of it as hiring a **team of security guards** to check your house. Each guard specializes in something different — one checks locks, another windows, another alarms. They go room by room and give you a report.

**Cyber Neo does that with your code**. Claude deploys **5 subagents working at the same time**, each one reviewing a different security area. At the end, you get a **Markdown report on your Desktop** with everything sorted by priority.

The most important part: **100% read-only**. It **never modifies, deletes, or changes** any file. It just looks and tells you what it sees.

Repo: `Hainrixz/cyber-neo`. MIT.

## How to use it

Inside Claude Code, in your project folder:

```
/cyber-neo
```

Claude:
1. Deploys 5 subagents in parallel
2. Each one reviews a specific domain
3. Finishes in 5-15 min depending on project size
4. **Saves the report to your Desktop** as markdown

...

## Low priority issues
...

Each issue with file, line, evidence, severity, **suggested fix**.

## Use cases

### 1. Pre-deploy to production

Before pushing to production:

```
> /cyber-neo
```

If there are criticals, **DON'T deploy** until you fix them.

### 2. Periodic audit of a project in production

Every month:

```
> /cyber-neo
```

Catch security drift without needing an expensive human auditor.

### 3. Pre-merge code review

Before merging a big PR:

```
> /cyber-neo --since main
```

It only reviews the PR's changes.

### 4. Onboarding to a new project

You inherit a codebase. Before touching anything:

```
> /cyber-neo
```

You see what security debt it carries **before** proposing changes.

### 5. Pitch to an investor / client

Show the Cyber Neo report to prove you **take security seriously**.

## Anti-patterns

### 1. Running it and not acting

Having the report isn't the same as **fixing the issues**. **Schedule** the fixes.

### 2. Trusting 100% in automated

Cyber Neo is an **automated audit**, not a human one. For critical production with strict compliance, **complement it with a professional human auditor**.

### 3. Only auditing pre-deploy once

Security **degrades over time**. Every deploy can introduce new issues. **Audit periodically**.

### 4. Ignoring the "low priority" ones

Today's lows are **tomorrow's criticals** if they pile up. Address even the lows.

### 5. Expecting a perfect audit

Cyber Neo covers OWASP + common standards. Vulnerabilities **specific to your business** or **business logic** (badly modeled authorization, payment race conditions) require a human eye.

## What it does NOT audit

To stay honest:

- ❌ Runtime vulnerabilities (you need DAST tools)
- ❌ Business logic flaws specific to your business
- ❌ Social engineering risks
- ❌ Physical security
- ❌ Infrastructure-as-Code beyond basics
- ❌ Deep container security

For those, a **professional human auditor**.

---

## When it's not a good fit

| Case | Why |
|------|---------|
| Regulated compliance (HIPAA, PCI-DSS) | You need a certified auditor |
| Critical enterprise product | Professional pentesting is better |
| Unsupported stack (Go, Rust, etc.) | Cyber Neo is JS/TS/Python only for now |
| You just want a "checkmark" without acting | It doesn't help |
