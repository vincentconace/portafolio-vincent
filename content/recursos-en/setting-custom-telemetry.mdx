## What it is

Configure how Claude Code handles telemetry:

- `CLAUDE_CODE_ENABLE_TELEMETRY=0` — disables the default
- `OTEL_METRICS_EXPORTER=custom` — uses a custom OTel exporter

Useful for:
- **Privacy:** disable telemetry entirely
- **Enterprise observability:** send it to your own Grafana/Datadog/Logfire
- **Compliance:** control where the data goes

## What you can do with it

### Privacy first

`CLAUDE_CODE_ENABLE_TELEMETRY=0` keeps usage info from going to Anthropic.

### Enterprise observability

You send telemetry to your own OTel pipeline. Your Dynatrace/Grafana/Logfire sees Claude Code metrics alongside everything else.

### Regional compliance

Some countries require metrics to stay in local data centers. A custom exporter makes that possible.

### Combo with [Logfire MCP](/boveda/mcp-logfire)

Logfire MCP for queries. Custom Telemetry to send metrics to Logfire. A natural combo.

### Combo with [Dynatrace MCP](/boveda/mcp-dynatrace)

If your stack is Dynatrace, same pattern.

## When it's NOT worth it

<Callout type="warning" title="Before you install">

- **For individual use:** you don't need serious observability.
- **Without a configured OTel pipeline:** a custom exporter needs a destination.
- **For Anthropic feature improvements:** disabling telemetry makes it harder for them to see patterns. A trade-off.
- **Only 32 installs:** enterprise niche. Validate the implementation.

</Callout>
