## Step 1 — The interview

You show up to a friend and tell them "I want to make an app." Your friend **does NOT start building**. First they ask you things.

### What Claude is going to ask you

- What do you want your app to do?
- Who is it for?
- How do you picture it visually?
- Are there similar apps you like?
- What's the MOST important thing that has to work?
- What's the main device? (web, mobile, both)
- How is your customer going to pay?
- What happens after they pay?

### Important tip

**More detail = better app**. Don't say "I want a store." Say:

> "I want a store where people can see my cakes, pick a flavor + size, and order via WhatsApp with prepayment through Mercado Pago."

### Prompt to start the interview

```
I want to build [APP / PAGE / TOOL — your idea].

Before doing anything, interview me. Ask me EVERYTHING you
need to know to be able to build it:

- What exactly do I want it to do?
- Who is it for?
- How do I want it to look?
- What features does it need?
- How do I charge?
- What's the main device?
- What happens after each important action?

Ask me the questions one by one. Wait for my answer before
the next one. If you need clarification, dig deeper.

At the end, summarize what you understood so I can confirm.
```

## Step 2 — The plan

With info from the interview, Claude builds an **organized map**: what gets built first, what comes after, all in order.

**Without a plan, you get lost**. The plan is your map.

### Prompt to generate the plan

```
Great. With everything we talked about, put together a build plan:

1. PRODUCT SUMMARY (5 lines)
   - What it is
   - Who it's for
   - How it makes money (if applicable)
   - Why someone would use it

2. FEATURES BY PRIORITY
   - MUST-have v1 (3-5 features)
   - NICE-have v2 (later)
   - DON'T DO v1 (what we explicitly leave out)

3. TECH STACK
   - Frontend
   - Backend
   - DB
   - Deploy
   - Payments (if applicable)
   - Justify each choice in 1 line

4. BUILD ORDER
   - Step 1, 2, 3... explained so Claude Code can follow it
   - For each step: what files to create, what it does

5. NEXT PRACTICAL STEP
   - What do I do TODAY to get started?

Save this as markdown. I'm going to use it in Claude Code.
```

## Step 3 — Build it

### Setup

1. Install Claude Code ([install Claude Code](/boveda/instalar-claude-code))
2. Create a folder for your project:

```bash
mkdir mi-app && cd mi-app && claude
```

3. Paste the plan you generated into `BLUEPRINT.md`

### Initial prompt

```
> Read BLUEPRINT.md.

  We're in Phase 1 MVP. Build the first step according to the
  order in the blueprint.

  Before coding, use Plan Mode (Shift+Tab).
  Tell me what files you're going to create, what dependencies to install.

  Wait for my OK to build.

  If something isn't clear, ask me BEFORE making it up.
```

### Iteration

When Claude finishes step 1:

```
> Let's test what you built. Run the local server and open
  the URL for me.

  I'm going to test it. I'll let you know if it works.
```

You test it. If something doesn't work:

```
> It doesn't work [DESCRIBE THE PROBLEM].
> Diagnose and fix it.
```

If it works:

```
> It works. Next step of the blueprint.
```

---

## The iteration cycle

```
1. Claude builds step N of the plan
2. You test it
3. If it works → next step
4. If not → you describe the problem → Claude fixes it
```

Each step is **small**, testable, reversible.

## Anti-patterns

### 1. Skipping the interview

If you go straight to Claude Code with "build me an app for [X]," **you lose the critical context**. Go through chat first.

### 2. Wanting Spotify in v1

Your first version does NOT need all the features of established products. **3-5 well-done features > 15 mediocre ones**.

### 3. Changing the stack midway

You started with Next.js → "SvelteKit would be better." **No**. Stick with it until v1.

### 4. Not validating between steps

If you build 5 steps without testing anything, when something breaks **you won't know where**. Test each step.

### 5. Buying domains before the MVP

Domain + hosting setup + paying for SaaS = $50-200 with nothing built. **Validate first**.

## When this approach is NOT a good fit

| Case | Why |
|------|---------|
| Super complex app (banking, healthcare) | You need real devs |
| No clear idea | Validate the idea first |
| No patience to iterate | It's a process, not instant |
| You expect your app to be ChatGPT v2 | Be realistic about v1 scope |
