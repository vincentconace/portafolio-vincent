## Qué es

Setting que apunta el statusline a un shell script custom (`~/.claude/statusline.sh` por default). El script recibe session context vía JSON stdin y devuelve la string a mostrar.

Para quienes querés full control: armás tu propio statusline con la info que sea relevante para vos.

JSON stdin incluye: model name, current directory, git info, etc.

## Cómo se instala

```bash
npx claude-code-templates@latest --setting statusline/command-statusline
```

Después editás `~/.claude/statusline.sh` con tu lógica.

## Qué podés hacer con esto

### Statusline propio

Combinás info que querés exactamente. Sin gimmicks ni decisión de otro.

### Para teams con preferencias específicas

Tu org puede tener convenciones específicas (ticket number en branch → mostrar en statusline). Custom script lo permite.

### Mix con tus tools

Si tenés health checks de tu app local, los podés meter en el statusline.

### Punto de partida sólido

Otros statuslines son shell scripts también. Podés tomar [Colorful Statusline](/boveda/setting-colorful-statusline) como base y modificar.

## Cuándo NO conviene

<Callout type="warning" title="Antes de instalar">

- **Para no-shell-comfortable:** mantenimiento del script puede ser fricción.
- **Si los statuslines existentes te alcanzan:** [Context Monitor](/boveda/setting-context-monitor) cubre 80% de casos.
- **Si tu script es lento:** statusline se renderiza seguido. Lento = experiencia laggy.
- **Sin testing:** un bug en el script puede romper statusline en silencio.

</Callout>
