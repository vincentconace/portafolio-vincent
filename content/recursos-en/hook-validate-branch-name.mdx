## What it is

Before any `git checkout -b` or `git switch -c`, it validates that the branch name follows Git Flow:
- `feature/<name>` for features
- `release/v*.*.*` for releases (semver)
- `hotfix/<name>` for hotfixes

If it doesn't match, it blocks the creation. Hook event: `PreToolUse`.

```

```

## What you can do with this

### Enforce Git Flow without discipline

You don't have to remember the convention — the hook enforces it.

### For teams where the branch name matters

CI scripts that match by prefix (`feature/*` triggers certain tests), automated release tooling — all of it needs the convention respected.

### For autonomous sessions

Claude doesn't invent random names — it has to follow the convention.

### Combo with [Prevent Direct Push](/boveda/hook-prevent-direct-push) and [Conventional Commits](/boveda/hook-conventional-commits)

Triple Git Flow gate:
1. Valid branch name
2. No direct push to main
3. Conventional commit format

## When it's NOT worth it

<Callout type="warning" title="Before installing">

- **If your team doesn't use Git Flow:** another convention might apply (trunk-based, etc.). Adjust the patterns or don't install it.
- **For personal / experimental repos:** overhead.
- **If your CI already validates:** this hook duplicates that, but locally.
- **Quick prototypes:** overriding it may be frequent.

</Callout>
