## El concepto antes que la lista

La diferencia entre "Claude prendido" y "Claude operando tu semana" es **tener skills bien armadas** para los trabajos que repetís. Una skill es **un archivo `SKILL.md`** en una carpeta — Claude lo lee y sabe el trabajo sin que se lo expliques otra vez.

> La trampa: la gente lee "40 skills" y siente que tiene que instalar 40. **No. Elegís 5 que matchean lo que hacés vos.** El resto es ruido.

Acá las dividimos en 6 categorías + un pack recomendado por perfil al final.

---

## Anatomía mínima de un SKILL.md

Todas las skills siguen el mismo formato:

```markdown
---
name: nombre-en-kebab
description: Una línea explicando cuándo Claude debe activar esto.
  Incluí palabras clave que el usuario va a usar naturalmente.
---

# Título de la skill

[Instrucciones que Claude sigue cuando la skill se activa]

## Protocolo
1. Primero hacé X
2. Después Y
3. Si pasa Z, hacé W

## Reglas
- NO hagas esto
- SIEMPRE confirmá esto antes de tal cosa

## Formato de salida
[Estructura del output esperado]
```

Las skills van en `~/.claude/skills/<nombre>/SKILL.md` (global) o `.claude/skills/<nombre>/SKILL.md` (por proyecto).

Más detalles en [anatomía de Claude](/boveda/anatomia-claude).

## Cómo elegís qué instalar

Aplicá este filtro:

1. **¿Esta tarea la hago al menos 1 vez por semana?** Si no, no necesitás skill — un prompt suelto alcanza.
2. **¿El output debería ser consistente entre veces?** Si sí, skill es mejor (asegura formato repetible).
3. **¿Quiero que Claude la active automáticamente cuando detecte el patrón?** Skills se activan solas por su descripción. Si querés invocación manual, un slash command alcanza.

Si las 3 son sí, hacé skill. Si no, ahorrá la complejidad.

## Anti-patrones

### 1. Copy-paste de 40 SKILL.md el día 1

Claude se confunde con 40 skills activas simultáneamente. Algunas se gatillan cuando no deberían. Empezá con **5**, mirá cómo van, sumá más cuando notes el gap.

### 2. Skills genéricas tipo "ayudame con X"

Si la skill es genérica, no aporta sobre el comportamiento default de Claude. Las skills útiles son **específicas a tu contexto** (tu negocio, tu codebase, tus convenciones).

### 3. Sin tunear el `description`

El `description` es lo que Claude lee para decidir cuándo activar. Si es vago, la skill no se gatilla. **Incluí palabras concretas** que vos vas a usar al pedirla naturalmente.

### 4. No iterar las skills

Una skill no es "se instala y queda". Las vas mejorando con uso. Si una skill da outputs mediocres, **ajustá el SKILL.md** — no tires la skill ni te resignás.

---

## La filosofía

Tener skills es como tener checklists profesionales. No reemplazan tu criterio — **encapsulan tu mejor versión del trabajo en formato repetible**. La primera vez que armás una skill bien, te lleva 30 minutos. Después la usás 200 veces.

El ROI no es la primera vez. Es las 200 veces que viene después.
