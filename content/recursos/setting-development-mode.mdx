## Qué es

Setting "all-in-one" para active development. Combina:
- **Allow:** 18 operaciones comunes (git, npm, build, test, lint, fs reads, etc.)
- **Deny:** 4 sensitive paths (.env, secrets, etc.)

En vez de instalar [Allow Git Operations](/boveda/setting-allow-git-operations) + [Allow Npm Commands](/boveda/setting-allow-npm-commands) + [Deny Sensitive Files](/boveda/setting-deny-sensitive-files) por separado, este los junta.

1,983 installs lo posicionan como uno de los settings de permissions más adoptados.

## Cómo se instala

```bash
npx claude-code-templates@latest --setting permissions/development-mode
```

Para Claude:

```
Instalé el setting permissions/development-mode. Listame las 18 ops
allowed y los 4 paths denied. Mostrame cómo agregar más allow si
necesito.
```

## Qué podés hacer con esto

### Setup permissions de un comando

En vez de juntar 3 settings, uno los cubre.

### Trusted dev environments

Si tu repo y máquina son seguros, el preset balanceado te da workflow sin fricción.

### Combo con hooks de seguridad

Settings establecen baseline. Hooks ([Dangerous Command Blocker](/boveda/hook-dangerous-command-blocker), [AI Bash Guard](/boveda/hook-ai-bash-guard)) agregan capas dinámicas.

### Para sesiones autónomas

`/goal` largos no se traban pidiendo permission por cada step.

## Cuándo NO conviene

<Callout type="warning" title="Antes de instalar">

- **Para repos prod o sensibles:** el set amplio puede ser demasiado permisivo. Custom config recomendado.
- **Si ya tenés [Allow Git/Npm](/boveda/setting-allow-git-operations) por separado:** no instales este encima — duplicación.
- **Si no confiás en tu environment:** ajustá scope antes.

</Callout>
