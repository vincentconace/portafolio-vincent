## What it is

Gordon Mickel built a marketplace with two plugins (`flow-next` and `flow`) focused on one very concrete thing: **reliable AI coding**. The narrative he pursues is "less magic, more predictability" — and to achieve it he combines four mechanisms:

1. **Plan-first** — before touching code, define what you're going to do.
2. **Ralph autonomous mode** — overnight coding with fresh context (the Ralph Wiggum technique: iterating over the same problem with fresh eyes).
3. **Multi-model review gates** — gates that only get crossed if Codex or RepoPrompt approve.
4. **Re-anchoring** — the plugin re-orients Claude back to the original spec when it detects drift.

`flow-next` (16 skills, 20 agents) is the current version; `flow` (7 skills, 6 agents) is the simple version. If you're just starting out, recommended: `flow`. If you want the full arsenal: `flow-next`.

## What you can do with it

### Plan-first without it breaking during execution

Flow forces you to plan before touching code. The difference from [native /plan mode](/boveda/goal-claude-code) or [Spec Kit](/boveda/spec-kit) is that the plan persists in `.flow/` and execution is then tied to that plan. If Claude starts to drift, the re-anchoring hook brings it back.

### Ralph autonomous mode — overnight coding with context resets

Ralph is a technique where Claude iterates over the same problem repeatedly, seeing its previous work with fresh context. Each iteration resets the context but sees the prior outputs. It helps prevent it from getting stuck on a bad initial decision. Compare with Anthropic's official `ralph-loop` in [Claude Plugins Official](/boveda/plugin-claude-plugins-official).

### Multi-model review gates

The gates won't let you move to the next step unless the plugin receives approval from a second model (Codex or RepoPrompt). Similar to [Claude Review Loop](/boveda/plugin-claude-review-loop) but as a gate, not an infinite loop.

### Worker subagent per task (in flow-next)

Each task runs in an isolated subagent, with its own context. This prevents parallel tasks from stepping on each other — useful when you launch several features at once.

### Re-anchoring against drift

When a long session starts drifting from the original plan ("Claude is implementing something else"), the plugin detects the drift and re-centers. It's the most typical problem of long autonomous sessions.

## Real benefits

- **Reliable coding, not magic** — the author's central promise.
- **Ralph mode with context reset** — the key difference from loops that keep the context.
- **Multi-model gates with Codex** — leverages your sub if you have it.
- **Worker subagents** — parallelism without collisions.
- **Its own site with docs** ([mickel.tech/apps/flow-next](https://mickel.tech/apps/flow-next)).

## When it's NOT worth it

<Callout type="warning" title="Before installing">

- **If you don't want "plan-first":** Flow forces you to plan. If you want to jump straight to code, you'll hit friction.
- **Without Codex / RepoPrompt:** the multi-model gates lose value. It still works, but half the value is in that.
- **Ralph mode burns tokens seriously:** iterating many times over the same problem multiplies the cost. For small tasks it doesn't add much.
- **Confusion flow vs flow-next:** they're two different plugins. If you install both, you'll get duplicate slash commands.
- **Not for beginners:** the learning curve is serious. Start with simple `flow` before jumping to `flow-next`.

</Callout>
