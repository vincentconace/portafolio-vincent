## What it is

A setting that configures `ANTHROPIC_CUSTOM_HEADERS` to add custom HTTP headers to every request to the API. Useful for:

- **Tenant identification** (`X-Company-ID: abc`)
- **Environment routing** (`X-Environment: production`)
- **API versioning** (`X-API-Version: v1`)
- **Custom auth schemes**

## How to install it

```bash
npx claude-code-templates@latest --setting api/custom-headers
```

Set up your env:

```bash
export ANTHROPIC_CUSTOM_HEADERS="X-Company-ID: your-company-id
X-Environment: production
X-API-Version: v1"
```

For Claude:

```
I installed the api/custom-headers setting. Verify that the headers are
sent correctly with a test request to the API.
```

## What you can do with this

### Multi-tenant SaaS

If your org uses the Anthropic API in a multi-tenant setup, the headers let you distinguish tenants.

### Routing by env

Your proxy can route based on headers. `X-Environment: staging` takes you to the staging endpoint.

### Custom auth gateway

If your org has its own API gateway in front of the real API, the headers let the gateway validate.

### Combo with [Corporate Proxy](/boveda/setting-corporate-proxy)

Proxy + Custom Headers = complete enterprise networking.

### Combo with [Bedrock Configuration](/boveda/setting-bedrock-configuration)

If you use Bedrock with custom auth, headers complement it.

## When it's NOT worth it

<Callout type="warning" title="Before you install">

- **For individual use:** it adds nothing.
- **If your API doesn't expect custom headers:** they may break requests.
- **For simple auth with an API key:** [API Key Helper](/boveda/setting-api-key-helper) or a direct env var is enough.
- **Without coordinating with the backend team:** the headers have to match what the server expects.

</Callout>
