## Qué es

Browserbase es una infraestructura cloud de browsers headless, mantenida como producto enterprise. Stagehand es el framework de browser automation que combina LLM con primitives tradicionales (click, type, navigate) para que sea más robusto a cambios en la página.

Este MCP los une: cloud browsers de Browserbase + agente de Stagehand controlados desde Claude Code. Vos no corrés Chrome local; Browserbase te da uno en cloud. Stagehand orquesta las acciones.

La gran ventaja vs [browser-use](/boveda/mcp-browser-use-mcp-server) o [Chrome DevTools](/boveda/mcp-chrome-devtools): no consume tu RAM, escala a paralelos, y maneja anti-bot mejor por ser tráfico residencial real (depende del plan).

## Qué podés hacer con esto

### Automation sin instalar nada local

"Andá a `tu-app.com/checkout` en el cloud browser, completá el form con datos de prueba, observá el flow y traeme screenshot al final". Sin que tu Mac transpire con Chrome.

### Paralelismo

"Abrí 10 sesiones de browser en paralelo y testeame cada uno con un set distinto de credenciales". Browserbase escala. Local no.

### Anti-bot serio

Si Cloudflare o DataDome te bloquean con Playwright local, Browserbase (según plan) ofrece browsers más realistas / residenciales.

### Comparación con alternativas

- **vs [browser-use](/boveda/mcp-browser-use-mcp-server):** browser-use es Python + local + open source. Browserbase es cloud + comercial + más robusto para producción.
- **vs [Chrome DevTools](/boveda/mcp-chrome-devtools):** CDP local es para debugging. Browserbase es para automation productiva a escala.
- **vs [Playwright Skill](/boveda/plugin-playwright-skill):** Playwright Skill es local + model-invoked. Browserbase es cloud + más opinionado.

### Stagehand como capa de inteligencia

Stagehand no es solo wrapper de Playwright. Es un sistema que combina LLM (para entender la página y decidir acciones) con primitivas determinísticas (las ejecuciones reales). Resulta más robusto cuando el HTML cambia.

## Cuándo NO conviene

<Callout type="warning" title="Antes de instalar">

- **Sin presupuesto:** Browserbase es comercial. Sin plan, no funciona. Para indie/hobby, browser-use local.
- **Para debugging interactivo:** browsers cloud son menos cómodos para devtools. Usá local.
- **Latencia:** cloud browsers tienen RTT mayor que local. Para feedback rápido (microflows), local gana.
- **Costos a escala:** paralelismo cuesta. Mirá pricing antes de batchear 1000 sesiones.
- **Compliance:** algunos datos no pueden salir de tu infra. Browserbase es cloud — verificá tu policy.

</Callout>
