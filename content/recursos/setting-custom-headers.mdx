## Qué es

Setting que configura `ANTHROPIC_CUSTOM_HEADERS` para agregar headers HTTP custom a cada request hacia el API. Útil para:

- **Tenant identification** (`X-Company-ID: abc`)
- **Environment routing** (`X-Environment: production`)
- **API versioning** (`X-API-Version: v1`)
- **Custom auth schemes**

## Cómo se instala

```bash
npx claude-code-templates@latest --setting api/custom-headers
```

Configurá tu env:

```bash
export ANTHROPIC_CUSTOM_HEADERS="X-Company-ID: tu-company-id
X-Environment: production
X-API-Version: v1"
```

Para Claude:

```
Instalé el setting api/custom-headers. Verificá que los headers se
mandan correctamente con una test request al API.
```

## Qué podés hacer con esto

### Multi-tenant SaaS

Si tu org usa Anthropic API en multi-tenant, los headers permiten distinguir tenants.

### Routing por env

Tu proxy puede routear según headers. `X-Environment: staging` te lleva al staging endpoint.

### Custom auth gateway

Si tu org tiene API gateway propio antes del API real, los headers permiten que el gateway valide.

### Combo con [Corporate Proxy](/boveda/setting-corporate-proxy)

Proxy + Custom Headers = enterprise networking completo.

### Combo con [Bedrock Configuration](/boveda/setting-bedrock-configuration)

Si usás Bedrock con custom auth, headers complementan.

## Cuándo NO conviene

<Callout type="warning" title="Antes de instalar">

- **Para uso individual:** no aporta.
- **Si tu API no espera headers custom:** pueden romper requests.
- **Para auth simple con API key:** [API Key Helper](/boveda/setting-api-key-helper) o env var direct alcanza.
- **Sin coord con backend team:** los headers tienen que matchear lo que espera el server.

</Callout>
