## Truco 1 — Ultra Think + Plan Mode

### Ultra Think

Claude tiene **2 modos** de pensar:
- **Rápido** (preguntas obvias)
- **Profundo** (problemas complicados paso a paso)

`ultrathink` activa el modo profundo. Es decirle: **"No te apures, pensalo bien"**.

### Plan Mode

Modo **arquitecto**. En lugar de empezar a codear, Claude:
1. Investiga tu proyecto
2. Entiende la estructura
3. Crea plan detallado **antes** de tocar nada

Activar: `/plan` o `Shift + Tab`.

### El combo

Plan Mode + ultrathink = **plan mucho más completo**, menos errores, mejor estructura.

Como pedirle a arquitecto que no solo dibuje planos, sino que **los revise 3 veces** antes de dártelos.

### Prompt

```
Activá Plan Mode (Shift+Tab).

ultrathink — Necesito que analices mi proyecto completo y crees
plan detallado para [DESCRIBÍ TAREA].

Investigá la estructura actual, identificá todos los archivos
que necesitás tocar, y dame plan paso a paso ANTES de escribir
cualquier código.
```

### Aviso 2026

`ultrathink` SIGUE funcionando. Pero "think hard", "think harder", "think more" **YA NO** (se interpretan como texto normal). Ver [guía Opus 4.7](/boveda/guia-claude-opus-47).

## Truco 2 — Sub-agentes

### Qué son

Imaginate proyecto grande (construir casa). Podés tener UN trabajador haciendo todo (plomería + eléctrica + paredes + techo), o **equipo de especialistas** cada uno en lo suyo.

Sub-agentes son **trabajadores separados** que Claude crea para atacar partes distintas **en paralelo**.

### Por qué son mejores

- **Velocidad**: en lugar de secuencial, paralelo. Tarea de 4 partes → potencialmente **4x más rápido**
- **Menos errores**: cada sub-agente foco en UNA cosa. Sin confusión de contexto
- **Menos contexto consumido**: cada uno tiene su propio scope

### Cuándo brilla

- Refactor de proyectos grandes
- Implementar features que tocan front + back + DB + tests
- Trabajos donde podés dividir en partes independientes

### Prompt

```
Acá hay tarea grande: [TAREA].

Dividila en sub-agentes paralelos. Cada uno con su contexto
limpio. Sugerencia: 3-5 sub-agentes, no más.

Para cada sub-agente:
- Qué hace
- Qué archivos toca
- Qué outputs entrega

Después los lanzás en paralelo. Reportame al final cuando todos
terminen.
```

Ver [crear agentes con Claude](/boveda/crea-agentes-con-claude) para profundizar.

## Truco 3 — `/init`

### Qué hace

Genera `CLAUDE.md` inicial para tu proyecto. Claude:
1. Investiga el proyecto
2. Detecta stack, convenciones
3. Genera doc explicando cómo trabajar con este proyecto

### Cuándo usar

- **Proyecto nuevo**: lo primero que corrés después de inicializar
- **Proyecto heredado**: para que Claude entienda qué hay
- **Onboarding de teammate**: documentación auto-generada

### Cómo

Dentro de Claude Code:

```
/init
```

Te genera `CLAUDE.md` en la raíz con:
- Tech stack detectado
- Estructura de carpetas
- Convenciones de naming
- Scripts disponibles
- Comandos de test/build/deploy

### El truco

Después de `/init`, **personalizalo** (ver truco 4). El default es buen baseline pero hay que adaptarlo a tu workflow.

## Truco 4 — Personalizar CLAUDE.md

### Por qué importa

Claude **lee CLAUDE.md** al abrir tu proyecto. Lo que ponés ahí, **lo aplica siempre** sin que se lo recuerdes.

Es la mejor inversión de tiempo en Claude Code.

### Las 2 reglas estrella

#### Regla 1 — Verificar trabajo

```markdown
## Verificación

Después de cada cambio significativo:
1. Corré `npm test` (o el comando equivalente)
2. Corré `npm run build` (o equivalente)
3. Si algo falla, **arreglalo antes de declarar listo**
4. NO me digas "está listo" si los tests no pasan
```

Claude antes de decirte "listo" **verifica que efectivamente funcione**.

#### Regla 2 — Memoria entre sesiones

```markdown
## Memoria

Al terminar cada sesión, guardá en /docs/session-notes/[FECHA].md:
- Qué se hizo
- Decisiones importantes
- Bloqueos encontrados
- Próximos pasos

Al inicio de cada sesión, leé el último archivo en
/docs/session-notes/ para retomar contexto.
```

Claude **mantiene memoria** entre sesiones.

### Otras reglas útiles

```markdown
## Convenciones

- Estilo de código: Prettier defaults
- Naming: camelCase para variables, PascalCase para clases
- Comments: solo cuando el "por qué" no es obvio del código

## Limitaciones

- NO instales libs nuevas sin preguntarme primero
- NO modifiques archivos en /legacy/ (deprecated)
- NO commitees `.env`

## Tone

- Respondé en español
- Bullets, no párrafos largos
- Sin emojis en docs formales
```

### CLAUDE.md global

`~/.claude/CLAUDE.md` aplica a **todos los proyectos**. Útil para preferencias personales que no dependen del proyecto.

CLAUDE.md por proyecto override config global cuando aplica.

## Anti-patrones

### 1. CLAUDE.md de 500 líneas

Después de cierto tamaño, Claude **ignora partes**. Mantenelo en 50-150 líneas. Lo crítico, no todo.

### 2. Ultra Think para tareas chicas

`ultrathink + arregla este typo` = overhead innecesario. Solo para razonamiento profundo.

### 3. Sub-agentes para todo

Para tareas con pocos archivos, sub-agentes meten overhead. Útiles cuando hay **paralelismo real posible**.

### 4. No actualizar CLAUDE.md

Tu proyecto evoluciona. **Cada 2-3 meses revisá** CLAUDE.md. Borrar reglas obsoletas, agregar nuevas.

### 5. Olvidar que `/init` es solo el comienzo

`/init` genera baseline. Si no personalizás, **no aprovechás el truco**.
