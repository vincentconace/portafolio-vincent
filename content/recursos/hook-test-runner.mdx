## Qué es

Otro hook de testing post-edit. La diferencia con [Run Tests After Changes](/boveda/hook-run-tests-after-changes): detecta inteligentemente qué tests son relevantes según extensiones y file structure, en vez de correr toda la suite.

Hook event: `PostToolUse`.

| Mecanismo | Detect test files relevantes |

## Cómo se instala

```bash
npx claude-code-templates@latest --hook testing/test-runner
```

Para Claude:

```
Instalé el hook testing/test-runner. Editá `src/auth/login.ts` —
debería correr `tests/auth/login.test.ts` y dependencias, no toda
la suite.
```

## Qué podés hacer con esto

### Tests relevantes solo

En lugar de correr 1,000 tests por un edit chico, corre los 5 relacionados.

### Feedback más rápido

Suite parcial = segundos vs minutos.

### Combo con [TDD Gate](/boveda/hook-tdd-gate)

TDD Gate enforce que existan tests. Test Runner los ejecuta. Workflow TDD completo.

### Combo con [Lint On Save](/boveda/hook-lint-on-save)

Lint + tests relevantes = quality pipeline rápido.

## Cuándo elegir cuál

| Hook | Cuándo |
| --- | --- |
| **Test Runner (este)** | Querés tests inteligentes (relevantes only) |
| [Run Tests After Changes](/boveda/hook-run-tests-after-changes) | Querés suite completa (más adoptado, 1,017 installs) |
| Ambos | No los instales juntos |

## Cuándo NO conviene

<Callout type="warning" title="Antes de instalar">

- **Si tu file structure no tiene convention test/source clara:** el hook puede no detectar bien.
- **Para sesiones autónomas largas:** suite completa periódica puede ser más confiable.
- **Si tu suite es rápida:** correr todo es overhead bajo.

</Callout>
