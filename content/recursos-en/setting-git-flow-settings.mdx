## What it is

An all-in-one configuration for teams that adopt Git Flow seriously. It combines:

- **Statusline** custom display showing Git Flow status in real time
- **Permissions** (allow 21 + deny 6) tuned for Git Flow
- **Env vars** that define prefixes and branch names:
  - `GIT_FLOW_MAIN_BRANCH: main`
  - `GIT_FLOW_DEVELOP_BRANCH: develop`
  - `GIT_FLOW_PREFIX_FEATURE: feature/`
  - `GIT_FLOW_PREFIX_RELEASE: release/`
  - `GIT_FLOW_PREFIX_HOTFIX: hotfix/`
  - `GIT_FLOW_VERSION_TAG_PREFIX: v`
- **PreToolUse hooks** (3 hooks) for workflow enforcement

It's the "ready-made" option to enforce a consistent Git Flow.

## What you can do with this

### Consistent Git Flow enforcement

A single setting enforces ALL aspects: naming, permissions, statusline awareness.

### Onboarding teams to Git Flow

Instead of explaining 4 separate things, you install this setting and everyone uses it.

### Statusline with Git Flow context

You can see which type of branch you're on (feature/release/hotfix) and whether you're in compliance.

### Combo with individual hooks

If you want more control, you can disable the bundle and use individual hooks:
- [Validate Branch Name](/boveda/hook-validate-branch-name)
- [Prevent Direct Push](/boveda/hook-prevent-direct-push)
- [Conventional Commits](/boveda/hook-conventional-commits)

### Compliance audit

If your org requires Git Flow for releases, this setting enforces it automatically.

## When it's NOT a good fit

<Callout type="warning" title="Before installing">

- **If your team doesn't use Git Flow:** another workflow (trunk-based, GitHub Flow) needs a different setting.
- **If you want more granularity:** individual hooks give more control than the bundle.
- **If your branch names are different:** adjust the env vars first.
- **For personal / experimental repos:** Git Flow is overhead.

</Callout>
