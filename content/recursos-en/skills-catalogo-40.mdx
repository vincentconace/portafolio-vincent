## The concept before the list

The difference between "Claude turned on" and "Claude running your week" is **having well-built skills** for the jobs you repeat. A skill is **a `SKILL.md` file** in a folder — Claude reads it and knows the job without you having to explain it again.

> The trap: people read "40 skills" and feel like they have to install all 40. **No. You pick 5 that match what you do.** The rest is noise.

Here we split them into 6 categories + a recommended pack per profile at the end.

---

## Minimal anatomy of a SKILL.md

Every skill follows the same format:

```markdown
---
name: name-in-kebab
description: One line explaining when Claude should activate this.
  Include keywords the user will naturally use.
---

# Skill title

[Instructions Claude follows when the skill activates]

## Protocol
1. First do X
2. Then Y
3. If Z happens, do W

## Rules
- DON'T do this
- ALWAYS confirm this before that

## Output format
[Structure of the expected output]
```

Skills go in `~/.claude/skills/<name>/SKILL.md` (global) or `.claude/skills/<name>/SKILL.md` (per project).

More details in [Claude anatomy](/boveda/anatomia-claude).

## How to choose what to install

Apply this filter:

1. **Do I do this task at least once a week?** If not, you don't need a skill — a one-off prompt is enough.
2. **Should the output be consistent across runs?** If yes, a skill is better (it ensures a repeatable format).
3. **Do I want Claude to activate it automatically when it detects the pattern?** Skills activate on their own based on their description. If you want manual invocation, a slash command is enough.

If all 3 are yes, make a skill. If not, save yourself the complexity.

## Anti-patterns

### 1. Copy-pasting 40 SKILL.md files on day 1

Claude gets confused with 40 skills active at the same time. Some trigger when they shouldn't. Start with **5**, see how they go, add more when you notice the gap.

### 2. Generic skills like "help me with X"

If the skill is generic, it adds nothing over Claude's default behavior. Useful skills are **specific to your context** (your business, your codebase, your conventions).

### 3. Not tuning the `description`

The `description` is what Claude reads to decide when to activate. If it's vague, the skill won't trigger. **Include concrete words** you'll use when asking for it naturally.

### 4. Not iterating on your skills

A skill isn't "install it and it's done." You improve them with use. If a skill produces mediocre outputs, **adjust the SKILL.md** — don't throw the skill away or resign yourself to it.

---

## The philosophy

Having skills is like having professional checklists. They don't replace your judgment — **they encapsulate your best version of the job in a repeatable format**. The first time you build a skill well, it takes you 30 minutes. After that you use it 200 times.

The ROI isn't the first time. It's the 200 times that come after.
