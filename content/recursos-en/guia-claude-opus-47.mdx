## The problem

You get excited about the new **Opus 4.7**. You switch it on, write 3-4 prompts, and up pops the message:

> "You've hit your usage limit. Come back in 5 hours."

Your whole day, gone.

Why does this happen? Anthropic set Opus 4.7 to **think harder** by default. Every prompt eats **twice as much** of your plan. You don't notice until the message is already there.

This guide is the map so it **doesn't happen to you**.

## The 3 models as employees

Think of them as 3 employees on your team:

| Model | Personality | Cost |
|--------|--------------|-------|
| **Sonnet 4.6** | Your everyday worker. Fast, efficient, does 80% of the work without complaining | Barely costs you anything |
| **Opus 4.6** | The one who really thinks. Deep analysis, strategy, mid-sized code | Spends nicely but won't bankrupt you |
| **Opus 4.7** | Expensive genius. Long-horizon code, heavy agents, 3x better vision | Eats **twice** as much |

### Anthropic's official numbers

- **Code**: Opus 4.7 writes 13% better than 4.6 (CursorBench: 70% vs 58%)
- **Vision**: reads images **3x better** (XBOW: 98.5% vs 54.5%)
- **Instructions**: follows them more literally — the good and the bad (it no longer assumes what you didn't say)

### API costs (if you use the API instead of a subscription)

| Model | Input ($/1M tokens) | Output ($/1M tokens) |
|--------|---------------------|------------------------|
| Opus 4.7 | $5 | $25 |
| Opus 4.6 | $4 | $20 |
| Sonnet 4.6 | $3 | $15 |

On the API there's no 5-hour limit message — you pay for what you use.

## 6 prompting rules (from Anthropic's official docs)

They apply equally to Sonnet 4.6, Opus 4.6, and Opus 4.7. If you write prompts like this, you'll save half your plan.

### 1. Be clear and direct

Think of it like talking to a **brilliant employee who has no context** about your business. The more specific, the better.

❌ "Build me a dashboard"
✅ "Build me a sales dashboard with a monthly filter, a line chart on top and a product table below"

### 2. Give it the why

❌ "Don't use ellipses"
✅ "Don't use ellipses because this will be read aloud and ellipses don't get pronounced well"

Claude **generalizes from the why** and applies the rule even in cases you didn't think of.

### 3. Give it examples

2-3 well-crafted examples are worth more than 1000 words of instructions. Wrap them in tags:

```
<example>
Input: ...
Output: ...
</example>
```

Claude tells examples apart from the rest of the prompt.

### 4. Tell it WHAT TO DO, not what NOT to do

❌ "Don't use markdown"
✅ "Write in flowing prose with complete paragraphs"

Claude responds **better to positive instructions** than to negations.

### 5. Give it a role

One line at the start:

```
You are a Python expert who explains things to beginners
```

It changes the tone and the depth. **It's the cheapest thing to do and the most impactful.**

### 6. If it's something simple, lower the effort

By default Opus 4.7 thinks at **xhigh** level (the maximum). If you ask it for something short or routine:

```
Use effort medium for this task (not xhigh).
```

Don't burn extra tokens.

## 4 template prompts (in English on purpose)

These are in **English** because that's how they come from the official docs and that's how they perform best (training). Copy and paste them.

### 1. Short responses (anti-verbosity)

When Claude goes on too long:

```
Provide concise, focused responses. Skip non-essential context,
and keep examples minimal.
```

### 2. Confirm before risky actions

For coding agents — so it won't delete files, run `git push --force`, or push changes without asking:

```
Consider the reversibility and potential impact of your actions.
You are encouraged to take local, reversible actions like editing
files or running tests, but for actions that are hard to reverse,
affect shared systems, or could be destructive, ask the user
before proceeding.
```

### 3. No over-engineering

So it won't invent files, helpers, or abstractions you didn't ask for:

```
Avoid over-engineering. Only make changes that are directly
requested or clearly necessary. Keep solutions simple and focused.
Don't add features, refactor code, or make improvements beyond
what was asked.
```

### 4. Investigate before answering

Against code hallucinations — it forces it to **read the real files** before making claims:

```
Never speculate about code you have not opened. If the user
references a specific file, you MUST read the file before
answering. Make sure to investigate and read relevant files
BEFORE answering questions about the codebase.
```

## Anti-patterns specific to Opus 4.7

### 1. Turning it on "just in case"

If you set Opus 4.7 as your default, **every prompt** eats 2x. In 4-5 prompts on Pro $20, hello limit message.

**Default: Sonnet 4.6.** Bump it up when you need to.

### 2. Using it for mechanical tasks

Reformatting code, writing simple docs, renaming variables → **Sonnet or Haiku are enough**. Opus 4.7 charges you genius rates to do junior work.

### 3. Long sessions without context management

Opus 4.7 thinks a lot **per turn**. If your session has 30 turns with Opus 4.7, you spent the equivalent of 60 turns on Sonnet.

For long sessions: **Sonnet 4.6** as the base + call in Opus 4.7 only for the critical moments.

### 4. Massive code generation

Generating 500 lines of boilerplate → Sonnet. Generating complex core logic of 50 lines → Opus 4.7.

### 5. Not lowering effort when it applies

If you ask for something simple in Opus 4.7 and don't say "low effort," it thinks at maximum anyway. **Be explicit** about the effort based on the task.
