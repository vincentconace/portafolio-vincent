## What hooks are (and why they matter)

Hooks are **shell commands that Claude Code runs automatically** at specific points in its lifecycle. They aren't actions Claude decides to take; they're guaranteed behaviors that the harness runs on your behalf.

The difference is key: if you ask Claude "run the tests after every change", it'll do it **sometimes**. If you set it up as a `PostToolUse` hook, it'll happen **every time**, without you asking.

---

## The 5 available events

| Event | When it fires | What it's for |
|--------|----------------|----------------|
| `SessionStart` | When you open Claude Code in a project | Load context, show a dashboard, read CI status |
| `PreToolUse` | Before running a tool (Edit, Bash, etc.) | Validation, blocking dangerous commands |
| `PostToolUse` | After a tool runs successfully | Format-on-save, regenerate types, reload dev server |
| `UserPromptSubmit` | When you send a message | Inject extra context, translate, parse custom commands |
| `Stop` | When Claude finishes its turn | Notifications, persist state, trigger CI |

## Base configuration

Hooks go in `~/.claude/settings.json` (global) or in `.claude/settings.json` (per project). The second option is the recommended one — it travels with your repo:

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          { "type": "command", "command": "npx prettier --write \"$CLAUDE_FILE_PATHS\"" }
        ]
      }
    ]
  }
}
```

<Callout type="info" title="Available environment variables">
Claude Code exposes variables you can use in your hooks:
- `$CLAUDE_FILE_PATHS` — files touched in the operation
- `$CLAUDE_TOOL_NAME` — name of the tool that ran
- `$CLAUDE_TOOL_INPUT_JSON` — the tool's full input
</Callout>

## Case 1 — Format on save

The most common one. After every Edit or Write, it formats the file automatically.

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "echo \"$CLAUDE_FILE_PATHS\" | xargs -n1 npx prettier --write 2>/dev/null || true"
          }
        ]
      }
    ]
  }
}
```

The `|| true` at the end matters: if Prettier fails (a file with no formatter), it won't break the session.

## Case 2 — Block dangerous commands

Prevent Claude from running things you don't want, without having to confirm them manually.

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "node .claude/hooks/block-dangerous.js"
          }
        ]
      }
    ]
  }
}
```

And the `.claude/hooks/block-dangerous.js` script:

```js
const input = JSON.parse(process.env.CLAUDE_TOOL_INPUT_JSON || '{}')
const cmd = input.command || ''

const banned = [
  /rm\s+-rf\s+\//,
  /git\s+push\s+--force.*main/,
  /DROP\s+TABLE/i,
  /:\(\)\{:\|:&\};:/,  // fork bomb
]

const hit = banned.find((re) => re.test(cmd))
if (hit) {
  console.error(`Command blocked by hook: matched ${hit}`)
  process.exit(2)  // exit code 2 = block
}
process.exit(0)
```

<Callout type="warning" title="Exit code = control">
A hook that exits with code `0` lets things continue. Code `2` **blocks** the operation. Any other code is treated as an error and shows the output to the user.
</Callout>

## Case 3 — SessionStart with CI context

When you open Claude Code in a repo, show it the status of the latest test run:

```json
{
  "hooks": {
    "SessionStart": [
      {
        "matcher": ".*",
        "hooks": [
          {
            "type": "command",
            "command": "gh run list --limit 1 --json status,conclusion,workflowName | jq -r '.[] | \"Latest CI: \\(.workflowName) → \\(.conclusion // .status)\"'"
          }
        ]
      }
    ]
  }
}
```

The hook's output is injected into the initial context. Claude sees it before replying to your first message.

## Case 4 — Notify on finish (Stop)

If you left Claude doing something long and you want it to ping you when it's done:

```json
{
  "hooks": {
    "Stop": [
      {
        "matcher": ".*",
        "hooks": [
          {
            "type": "command",
            "command": "osascript -e 'display notification \"Claude finished\" with title \"Claude Code\" sound name \"Glass\"'"
          }
        ]
      }
    ]
  }
}
```

On Linux use `notify-send`. On Windows, PowerShell with `New-BurntToastNotification`.

## Debugging hooks

If a hook doesn't run or throws a silent error:

```bash
claude --debug
```

It shows you every hook that fires, its exit code, stdout and stderr. It's the only sane way to iterate on complex hooks.

<Callout type="tip" title="Start simple">
The most common trap is writing an ambitious hook (5 things chained in a one-liner) that fails silently. Start with `echo "hook fired"` to confirm it fires, then add logic to it.
</Callout>
